## 1. 产品定位

面向 AI 硬件设备（摄像头 + 麦克风 + GPS）的智能衣橱助手。核心价值：帮助用户管理自己的衣橱，并在穿搭决策时提供 AI 辅助。

**设计原则**：
- **零摩擦**：所有自动化能做的事不让用户手动填，所有能延迟的决策不放在主流程里
- **Image first**：AI 说明文字默认极短（「更利落」「更适合今天温度」），长解释点开才展开；fashion 是视觉产品，不是阅读产品
- **Subtle AI 语气**：brief / tasteful / minimal，不做陪伴人格化，不输出废话式解释
- **用户可见文案规范**：只描述用户能得到什么，不写「不入库 / 不保留 / 不会触发 X」这种开发者解释。用户不关心系统不做什么。

|---|---|---|

| Color Analysis 色彩测试 | ❌ 推迟 | ✅ |
| Inspiration 灵感收藏 | ❌ | ✅ |

## 2. 产品层级架构

```
Level 0  全局 AI Hub
           所有 Skill 的对话入口，AI 主动感知并提供建议
           └─ Level 1  Fashion Agent（穿搭 Agent）
                         专属穿搭伙伴，管理衣橱、风格记忆与洞察
                         └─ Level 2  Styling 内联换装（首页卡片内展开）
                                       无独立页面，在首页推荐区内完成
```

- **AI Hub** 是设备级入口，Fashion Agent 是其中一个 Skill
- Hub 负责 Tomorrow Planning 推送（每晚 8 点）及天气/日历事件触发
- **Styling 不再是独立层级页面**，而是首页推荐卡片的内联交互状态
- **随手试穿**：用户向 AI Hub 发送任意外部图片（截图/种草图），AI 直接生成试穿效果图并以聊天卡片呈现，用户可选择入库或直接离开，**不强制导入**

对穿搭有一定要求、衣物数量较多、日常需要搭配决策的用户。

## 4. Onboarding

### 目标
首次启动到「衣橱可用」的步骤尽可能少。

### 流程

```
首次启动
  └─ 上传一张全身照（可跳过）
       └─ 直接进入 Fashion Agent 首页（空衣橱引导）
```

- 全身照用于后续试穿，非强制
- 跳过全身照时，换装时再引导上传，不在 onboarding 阻塞
- **不做**：性别/身材/风格偏好问卷（数据从行为推断）

## 5. Fashion Agent 首页 *(v1.1 主要变更)*

### 核心区块

- **今日穿搭推荐**：横向滑动展示 4-5 套穿搭效果图 + 每套附 AI 说明
- **风格洞察**：风格标签 pill 行，附月趋势箭头（↑/↓/—）
- **快捷入口**：Wardrobe（衣橱）/ Looks（穿搭日志）/ 添加单品
- **底部命令栏**（常驻）：自然语言输入框（语音/文字），「今天穿什么」「换条深色牛仔裤」「保存这套」等
- **底部分类栏**：内联换装激活时从底部滑入（不常驻）

> Color Analysis 快捷入口从首页移除（功能推迟），后续版本补回。

### 多套推荐展示（懒触发 + 状态机 + 前端轮询）

进首页**自动开始生图**，无需用户点 CTA：

- 元数据（推荐套数、风格标签、单品 ID 列表）秒返
- 效果图后台异步生成，前端定期轮询（约 8 秒一次）
- 命中已生成图 → 状态切换为 ready，替换占位
- 横向滑动浏览，默认停在第一套
- 「换一组」按钮可强制重算
- 每套展示：试穿效果图 + 风格标签 + 一句 AI 说明
- 用户点击任一套进入该套的内联换装状态

**状态反馈**（页面顶部 / 占位提示）：

| 状态 | 用户感受 |
|---|---|
| ready | 已展示效果图 |
| generating | 「正在为你生成今日效果图…」+ 单品 2×2 占位拼图先看着 |
| no_photo | 「先上传一张全身照，才能看到试穿效果」+ 上传 CTA |
| image2_offline | 「生图服务暂不可用，先看看搭配建议」（仍展示拼图） |
| no_wardrobe | 「衣橱还是空的，先添加几件单品吧」+ 上传 CTA |

### Styling 内联换装（取代独立 Styling Session 页面）

**不跳转新页面**，在首页推荐区直接展开：

```
用户点击某套穿搭效果图
  └─ 底部分类栏高亮激活
       └─ 用户点击某个分类（如「下装」）
            └─ 当前页面下方展开该类目衣橱单品列表（横向滚动）
                 └─ 点选某件单品 → 触发单张效果图重新生成
                      └─ 生成完成 → 效果图刷新，分类栏保持展开
                           └─ 用户点「完成」→ 可保存为 Look
```

**关键约束**：
- 换单品只重新生成当前这 1 张图，不重刷全部 4-5 套
- 生成中展示 loading 占位，其他套仍可滑动浏览
- 语音/文字输入（「换条深色牛仔裤」）也在此页面内响应，不跳转

**候选为空时的兜底**（衣橱里同品类只有当前那件）：
- 不假装替换成功，明确告知「你的衣橱里暂时只有这件 X，添加一件新的吧」
- 展示「去添加单品」CTA → 跳上传页
- 不调用生图（image2），避免用户等 5 分钟拿到一样的搭配

### 自然语言入口（底部命令栏）

用户在首页底部输入框（语音/文字）发送指令，**不跳转新页面**，回复在底部展开：

| 用户说 | 系统行为 |
|---|---|
| 「今天穿什么」「和朋友去玩穿什么」 | 临时回复 caption + 单品 4 件 2×2 缩略图；**不覆盖「今日推荐」卡片区**（两个语义独立） |
| 「换条裤子」「换件上衣」「上衣换成白色」 | 在当前展示的搭配上替换该品类的单品（候选空时走上述兜底） |
| 「保存这套」「记录今日穿搭」 | 保存当前展示的搭配到 Looks |
| 「我衣橱里有什么 X」 | 列出对应品类单品 |
| 其他 | 友好降级提示 |

**场合识别**：「和朋友去玩穿什么」「见教授穿什么」「约会穿什么」等场合表述会影响 caption 的措辞，但**不污染首页默认推荐缓存**（用户每次问不同场合 → 当次重算回复，下次不带场合的访问仍命中默认）。

### AI 主动推送（来自 Hub 层）

每天最多触发一次，同日取最高级：

| 级别 | 触发条件 | 示例文案 |
|---|---|---|
| S | 每晚 8 点 Tomorrow Planning | 「明天上海会下雨，你上午有 presentation，要不要现在搭一套？」 |
| A | 天气剧变（气温骤降 / 突然降雨） | 「明天突然降到 5°C，需要重新规划一下吗？」 |
| B | 日历检测到特殊事件 | 「后天有飞纽约的航班，要不要提前看看能搭什么？」 |

### Tomorrow Planning Session

**触发时间**：每晚 **20:00** 由设备本地后台任务触发（v1.1 实现为 FastAPI startup + asyncio 定时 loop，未来设备端可换 OS 级调度）。

```
20:00 触发预生成
  └─ 拿明日天气 → 推荐 + image2 拼图 + 裁切 → 写入次日缓存
       └─ 设备空闲时跑完（典型 1-5 分钟）
            └─ 每晚 20:00 Hub 推送（图已就绪，零额外等待）
                 └─ Hub 消息直接展示可横滑的 outfit 卡片（复用已生成图）
                      ├─ 用户在 Hub 层滑动浏览 4-5 套
                      ├─ 点某套 → 进入 Fashion Agent 做内联换装调整
                      └─ 用户确认 → 保存为「明日计划」
                           └─ 次日早晨提醒：「今天要穿昨晚那套吗？」
                                ├─ [穿这套] → 直接记录为当日 OOTD
                                └─ [重新搭] → 返回首页内联换装
```

**容错**：image2 离线 / 主照缺失 / 衣橱为空 → 跳过当晚预生成，次日首页走「懒触发」走自动上身流程。

> 💡 **待开会确认**：Hub 推送直接含可滑动 outfit 卡片（少一跳，视觉冲击强）vs 纯文字 CTA（点进 Fashion Agent 再看图）。前者要求 Hub 层支持横滑卡片交互，需确认是否符合 AI Hub 整体交互规范。

## 6. 衣橱（Wardrobe）

### 6.1 页面结构

- 顶部：总件数
- 分类 Tab：全部 / 上衣 / 下装 / 外套 / 鞋子 / 配饰 / 包包
- 内容区：单品网格，白底图 + 类型名

### 6.2 数据入口

| 入口 | 触发方式 | 适用场景 |
|---|---|---|
| 单品照上传 | 用户主动拍照或从相册选图 | 拿着一件衣服拍 |
| 截图导入 | 分享/截图进入 App | 购物网站、小红书种草图 |
| 穿搭照剥离 | 上传全身穿搭照，AI 自动提取各单品 | 整理旧照片入库 |

> 内部测试用数据通过 `import_batch.py` 批量写入，不作为用户侧功能暴露。

### 6.3 单品照上传流程

```
用户上传图片
  └─ AI 自动识别并打标（category / type / color / style / season / warmth / fit）
       └─ 展示识别结果，用户确认或修正 category / type
            └─ 点击「加入衣橱」→ 立即静默存入（原图）
                 └─ 后台异步 beautify：原图 → 干净白底平铺商品图
                      └─ 完成后自动替换衣橱列表里的展示图（前端轮询）
```

确认粒度：最少确认 category + type，其余 tag 自动存入。

**视觉识别 scope**：模型只识别**用户主动展示的主体单品**（占面积大、居中、平铺/挂起/手持），忽略背景货架陪衬、模糊遮挡、镜子反射、模特身上的其他衣物。识别不到主体时返回空，提示用户重新拍摄，**宁可漏识别也不要把背景当主体**。

**Beautify**：异步进行（image2 单次 1-5 分钟），衣橱列表先用原图占位，badge 显示「美化中」，完成后自动替换。若 image2 不可达则保留原图，不阻塞入库。

### 6.4 截图导入流程

与单品照上传流程一致。v1.1 直接送 GPT Vision 识别打标 + 用户确认，不做 inpaint 预处理。

> **截图导入 vs 随手试穿的区别**：导入 = 用户明确想将这件单品加入衣橱管理；随手试穿 = 用户只想看效果，不承诺入库。两者是不同意图，入口分开：导入走 Wardrobe 页，试穿走 AI Hub。

### 6.5 穿搭照剥离流程

```
用户上传全身穿搭照
  └─ AI 识别出 N 件单品，逐件剥离
       └─ 每件单品：
            ├─ 生成白底展示图（image_url）
            ├─ 保留原始 crop（image_crop_url，用于 CLIP 检索）
            └─ 自动打标 → 用户批量确认 category / type
```

### 6.6 图片数据结构

| 字段 | 内容 |
|---|---|
| `image_url` | 白底美化图，用于衣橱展示、分享 |
| `image_crop_url` | 原始裁剪图，用于 CLIP 特征提取 |

## 7. 多套穿搭生成技术方案 *(v1.1 新增)*

### 批量生图策略

**一次 image2 调用生成含多套穿搭的拼图，后端裁切得到独立效果图。**

```
推荐触发（首页加载 / Tomorrow Planning）
  └─ 构建批量 prompt：要求生成 N 套穿搭拼图（横排或竖排，约定固定格子数）
       └─ image2 返回单张拼图
            └─ 后端按固定坐标裁切 → N 张独立效果图
                 └─ 存入缓存，首页展示
```

**换单品局部刷新**：
```
用户换某一件单品
  └─ 只针对当前这套重新生成（单张，非拼图）
       └─ 生成完成后替换对应卡片
```

### Prompt 格式约定

拼图 prompt 需约定：
- 每套的排列位置（如「左右各一套」「2x2 网格」）
- 各套之间有足够间距，便于裁切
- 统一背景（studio / 纯色），不带文字水印

裁切坐标由 prompt 版本号对应的配置文件管理（见 `prompt_versions/`），prompt 变更时同步更新裁切配置。

## 8. AI Hub 随手试穿 *(v1.1 新增)*

**用途**：用户对外部单品（小红书截图、购物网站商品图等）产生兴趣时，随手发给 AI 看试穿效果，**不强制入库**。

### 触发方式
用户向 AI Hub 发送包含单品的图片。

### 流程

```
用户发图片到 AI Hub
  └─ AI 识别为外部单品（非 OOTD 全身照）
       └─ 接受任务，返回 task_id；后台异步生成（image2）
            └─ Hub 在结果就绪后推送聊天卡片：效果图 + 两个操作
                 ├─ [加入衣橱] → 走截图导入流程（Vision 打标 → 确认 → 入库）
                 └─ [不用了]   → 结束，图片不保留
```

### 异步契约（与 Tomorrow Planning / OOTD 等长任务共用）

所有"长任务 + 异步结果"模式（quick_tryon、Tomorrow Planning 预生成、未来 AI Look 事后生成）统一一套契约：

```
POST 提交  → 立即返回 {task_id, kind, status: queued}
GET  状态  → {task_id, kind, status: running | done | failed, input_url, result_url, error, created_at, completed_at}
```

**通知方式抽象**：当前 demo 用 polling（前端定期查询），将来 Hub 集成后改为 webhook（后端任务完成时主动调 `hub_callback_url`）——契约（task_id / status / result_url 字段）不变，只换通知方向。

### 与其他试穿入口的区分

| 入口 | 意图 | 单品来源 |
|---|---|---|
| AI Hub 随手试穿 | 探索，不承诺入库 | 外部图片（截图/种草） |
| 首页内联换装 | 从衣橱里换搭 | 已入库单品 |
| Wardrobe 截图导入 | 明确要入库管理 | 外部图片 |

## 9. 摄像头 OOTD 捕捉 *(v1.1 变更)*

**用途**：OOTD **记录**功能，用户已穿好，主动按硬件键记录今天穿了什么。

### 触发方式
用户主动按硬件快门键。（镜子自动检测留后续版本）

### 流程

```
用户按硬件快门键
  └─ 摄像头拍摄全身照
       └─ CLIP 闭集检索匹配衣橱单品
            └─ 展示匹配结果（匹配置信度低时给出 Top-3 候选）
                 └─ 用户确认或修正
                      └─ 发送聊天卡片到 AI Hub 对话流
                           └─ 用户点击卡片上「保存到 Looks」→ 归档
                                └─ 原始全身照不保留，只存单品 ID 列表
```

**v1.0 → v1.1 变化**：捕捉确认后不直接写入 Looks，先以聊天卡片形式呈现，由用户主动确认归档。原始照片不保存（隐私）。

## 10. Looks（穿搭日志）*(v1.1 变更)*

### 页面结构

- 顶部 Tab：全部 / 收藏 / 逛街 / 约会 / 通勤（场景分类）
- 内容区：穿搭图网格，附日期

### Look 数据来源
- 首页内联换装完成后保存
- OOTD 捕捉后用户确认归档（通过聊天卡片）
- 用户手动添加

### 图片存储策略 *(v1.1 变更)*

| 来源 | 图片保留策略 |
|---|---|
| Styling 生成的效果图 | 保留（AI 生成图，无隐私问题） |
| OOTD 摄像头拍摄原图 | **不保留**，只存单品 ID 列表 |
| Look 展示图 | OOTD 来源用单品白底图拼合；Styling 来源用 AI 效果图 |

### Look 卡片形态（两种，同一网格混排）

| 形态 | 来源 | 展示方式 | 尺寸 |
|---|---|---|---|
| 单品拼图卡 | OOTD 捕捉 | 白底单品图 2×2 拼合 | 标准 |

## 14. 技术方案

### 图像生成
- 服务：`image2`（HTTP API，`image2_client.py`）
- 首页推荐：1 次调用生成拼图 → 后端裁切
- 换单品局部刷新：1 次调用生成单张
- Prompt 版本与裁切坐标配置文件统一管理

### 衣橱检索
- 模型：**Marqo FashionSigLIP**（fine-tuned on 100万+ 时尚商品图）
- 备选：SigLIP 2（Google 2025）
- 场景：OOTD 捕捉匹配
- 特征来源：优先 `image_crop_url`，无则用 `image_url`
- 索引：FAISS，本地，衣橱变动时增量更新

### 天气
GPS 定位 → 天气 API → 气温区间 → 映射 `warmth` 字段

### 存储
本地 SQLite（`wardrobe.db`），当前阶段无需云同步

## 16. 产品形态与分发 *(v1.1 新增)*

**短期（v1.1）**：PWA（Progressive Web App）
- 同时支持 iOS Safari「添加到主屏幕」+ Android Chrome「安装」+ 桌面 Chrome / Edge「安装」
- 设备适配：safe-area 避让 iPhone 灵动岛 / 刘海 / Home Indicator；viewport-fit=cover
- 与最终硬件设备形态解耦：UI 层走 Web，业务协议层（`dispatch`）不变
- 暂不做 Service Worker 离线 / 推送（要 HTTPS + 复杂度高，留待真要离线时再上）

**最终形态**：取决于硬件方案
- 带触屏的家居设备 → 设备里跑浏览器壳，复用 PWA
- 仅摄像头 + 麦克风（无屏幕）→ 配套手机 App（PWA 或原生）
- 智能眼镜等极简 UI → 语音为主，UI 极简

**App / Web 分叉点**：在入口层（HTTP / native call）。`dispatch()` 协议层 `{action, payload, message}` 是稳定契约，未来无论换哪种形态，**业务模块零改动**。
