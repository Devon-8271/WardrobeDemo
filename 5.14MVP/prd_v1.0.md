# 智能衣橱助手 PRD v1.0

**版本**：1.0  
**日期**：2026-05-14  

---

## 1. 产品定位

面向 AI 硬件设备（摄像头 + 麦克风 + GPS）的智能衣橱助手。核心价值：帮助用户管理自己的衣橱，并在穿搭决策时提供 AI 辅助。

**设计原则**：
- **零摩擦**：所有自动化能做的事不让用户手动填，所有能延迟的决策不放在主流程里
- **Image first**：AI 说明文字默认极短（「更利落」「更适合今天温度」），长解释点开才展开；fashion 是视觉产品，不是阅读产品
- **Subtle AI 语气**：brief / tasteful / minimal，不做陪伴人格化，不输出废话式解释

### 版本范围（v1.0 vs 后续）

| 功能 | v1.0 | 后续版本 |
|---|---|---|
| 衣橱建模（上传 / 截图 / 穿搭照剥离） | ✅ | — |
| Styling Session（A+B=C 试穿） | ✅ | — |
| Looks 穿搭日志 | ✅ | — |
| Tomorrow Planning | ✅ | adaptive timing |
| 摄像头 OOTD 捕捉（硬件键触发） | ✅ | 镜子自动检测 |
| Style Identity 风格档案 | 弱化，仅首页展示风格标签 | 完整雷达图 + 趋势 |
| Color Analysis 色彩测试 | setup 入口（非首页一级） | 首页快捷入口 |
| Inspiration 灵感收藏 | ❌ | ✅ |

## 2. 产品层级架构

```
Level 0  全局 AI Hub
           所有 Skill 的对话入口，AI 主动感知并提供建议
           └─ Level 1  Fashion Agent（穿搭 Agent）
                         专属穿搭伙伴，管理衣橱、风格记忆与洞察
                         └─ Level 2  Styling Session（实时搭配模式）
                                       沉浸式搭配体验，AI 对话驱动试穿与调整
```

- **AI Hub** 是设备级入口，Fashion Agent 是其中一个 Skill
- Hub 负责 Tomorrow Planning 推送（每晚 8 点）及天气/日历事件触发
- Fashion Agent 内部的 Styling Session 是核心交互场景

## 3. 目标用户

对穿搭有一定要求、衣物数量较多、日常需要搭配决策的用户。

## 4. Onboarding

### 目标
首次启动到「衣橱可用」的步骤尽可能少。

### 流程

```
首次启动
  └─ 上传一张全身照（可跳过）
       └─ 直接进入 Fashion Agent 首页（空衣橱引导）
```

- 全身照用于后续试穿（Styling Session），非强制
- 跳过全身照时，试穿功能在使用时再引导上传，不在 onboarding 阻塞
- **不做**：性别/身材/风格偏好问卷（数据从行为推断，见第 10 节 Style Identity）

## 5. Fashion Agent 首页

用户进入 Fashion Agent 后看到的主界面。

### 核心区块

- **今日穿搭建议**：基于天气 + 衣橱自动生成一套推荐，附 AI 说明（「今天 12°C 微风，这套轻松又保暖」）
- **风格洞察**：展示当前主要风格标签及近期趋势（Clean Minimal ↑ / Neutral Tone ↑）
- **快捷入口**：Styling（实时搭配）/ Wardrobe（衣橱）/ Looks（穿搭日志）/ Color Analysis（色彩测试）

### AI 主动推送（来自 Hub 层）

**不做实时出门检测**（GPS 不稳定、误触发率高、体验 creepy）。推送基于以下三级优先级，每天最多触发一次，同日取最高级：

| 级别 | 触发条件 | 示例文案 |
|---|---|---|
| S | 每晚 8 点 Tomorrow Planning（固定时间，不开放自定义） | 「明天上海会下雨，你上午有 presentation，要不要现在搭一套？」 |
| A | 天气剧变（气温骤降 / 突然降雨） | 「明天突然降到 5°C，和今天差很多，需要重新规划一下吗？」 |
| B | 日历检测到特殊事件（面试 / 婚礼 / 出行） | 「后天有飞纽约的航班，要不要提前看看能搭什么？」 |

用户跳过后当天不再推送。点击后进入 **Tomorrow Planning Session**。

### Tomorrow Planning Session

```
Hub 推送（每晚 8 点）
  └─ 展示明天天气 + 日历事件摘要
       └─ AI 推荐一套搭配（试穿效果图）+ 理由说明
            ├─ 用户调整 → 多轮对话修改（流程同 Styling Session）
            └─ 用户确认 → 保存为「明日计划」
                 └─ 次日早晨提醒：「今天要穿昨晚那套吗？」
                      ├─ [穿这套] → 直接记录为当日 OOTD
                      └─ [重新搭] → 进入 Styling Session
```

用户也可主动告知计划触发 Planning：「明天要面试」「周末去海边」→ AI 进入 planning mode。

## 6. 衣橱（Wardrobe）

### 6.1 页面结构

- 顶部：总件数（共 N 件单品）
- 分类 Tab：全部 / 上衣 / 下装 / 外套 / 鞋子 / 配饰 / 包包
- 内容区：单品网格，展示白底图 + 类型名

### 6.2 数据入口（三种用户入口）

| 入口 | 触发方式 | 适用场景 |
|---|---|---|
| 单品照上传 | 用户主动拍照或从相册选图 | 拿着一件衣服拍 |
| 截图导入 | 分享/截图进入 App | 购物网站、小红书种草图 |
| 穿搭照剥离 | 上传全身穿搭照，AI 自动提取各单品 | 整理旧照片、上身照入库 |

> 内部测试用数据通过 `import_batch.py` 批量写入，不作为用户侧功能暴露。

### 6.3 单品照上传流程

```
用户上传图片
  └─ AI 自动识别并打标（category / type / color / style / season / warmth / fit）
       └─ 展示识别结果 + 缩略图，用户确认或修正 category / type
            └─ 点击「加入衣橱」→ 静默存入，其余 tag 可在详情页随时编辑
```

**确认粒度**：最少确认 category + type，其余 tag 自动存入不打扰用户。category 错误会导致下游推荐失效，必须确认。

### 6.4 截图导入流程

与单品照上传流程一致。截图可能含水印/文字/背景，v1.0 直接送 GPT Vision 识别打标 + 用户确认，不做 inpaint 预处理（留后续版本）。

### 6.5 穿搭照剥离流程

```
用户上传全身穿搭照
  └─ AI 识别出 N 件单品，逐件剥离
       └─ 每件单品：
            ├─ 生成白底展示图（image_url，用于衣橱展示）
            ├─ 保留原始 crop（image_crop_url，用于 CLIP 检索）
            └─ 自动打标 → 用户批量确认 category / type
                 └─ 全部确认后批量入库
```

**设计说明**：
- 白底图和原始 crop 均静默存储，用户只看到白底图
- 原始 crop 保留真实上身质感，CLIP 检索精度优于纯白底生成图
- 批量确认界面：横向滑动，每张图展示识别结果，用户单独纠正或直接「全部确认」

### 6.6 图片数据结构

| 字段 | 内容 |
|---|---|
| `image_url` | 白底美化图，用于衣橱展示、分享 |
| `image_crop_url` | 原始裁剪图，用于 CLIP 特征提取 |

## 7. Styling Session（实时搭配）

沉浸式搭配模式，AI 对话驱动，支持多轮调整。

### 7.1 触发方式
- 首页「去搭配」按钮
- Hub 主动推送后点击「去搭配」
- 摄像头检测到镜子场景（见第 8 节）

### 7.2 核心界面

进入后展示当前推荐搭配的试穿效果图（A+B=C），同时显示：
- 场景信息（天气 / 风格标签）
- AI 说明文字
- 快捷操作按钮（换裤子 / 更休闲一点 / 加一件外套 / 换鞋子）
- 底部分类栏（外套 / 上衣 / 下装 / 鞋子 / 包包 / 配饰）可直接浏览替换

### 7.3 AI 对话调整

用户通过语音或文字输入调整需求，AI 理解后更新试穿图：

```
用户：「换一条深色牛仔裤试试」
AI：重新生成试穿图 + 说明「这条直筒牛仔裤更显利落，也更符合 Clean Minimal 风格」
用户：「换双乐福鞋」
AI：更新鞋子 + 「乐福鞋让整体更优雅，需要我帮你看看袜子搭配吗？」
```

支持延伸建议：配饰建议 / 发型建议 / 优化整体建议。

### 7.4 试穿技术实现（A+B=C）

- **A**：衣橱单品图（`image_url`）
- **B**：用户全身照
- **C**：AI 生成上身效果图（`image2` 服务）
- **背景**：根据 style 标签自动选择（街头/运动 → 户外；正式/通勤 → 室内；其余 → studio），用户不选
- **版型**：默认「常规」，用户可选修身 / 宽松 / oversize

### 7.5 完成流程

```
用户点击「完成」
  └─ 保存当前搭配为 Look（可选命名场景）
       └─ 沉淀到 Looks 穿搭日志
```

## 8. 摄像头 OOTD 捕捉

> **注**：以下设计已定，图像质量与大模型识别能力待摄像头实测后验证。

**用途定位**：这是 OOTD **记录**功能，不是穿搭推荐。用户已穿好，主动按硬件键把今天穿了什么记下来，降低记录摩擦。

### 触发方式（v1.0）
用户主动按硬件快门键进入 OOTD 捕捉。

> 镜子自动检测（摄像头 CV）留后续版本，v1.0 不做——误触发率高，调试成本大。

### 流程

```
用户按硬件快门键
  └─ 摄像头拍摄全身照
       └─ CLIP 闭集检索匹配衣橱单品
            └─ 展示匹配结果，用户确认或修正
                 └─ 记录为当日 OOTD，沉淀到 Looks
```

### 设计说明
- 主动按键确保对焦准确，避免自动拍到模糊帧
- 检索范围限定在用户自己的衣橱（闭集）
- 匹配置信度低时展示 Top-3 候选，由用户点选确认

## 9. Looks（穿搭日志）

### 页面结构

- 顶部 Tab：全部 / 收藏 / 逛街 / 约会 / 通勤（场景分类）
- 内容区：穿搭图网格，附日期
- 场景统计卡片：约会 N 次搭配 / 运动 N 次 / 旅行 N 件单品等

### Look 数据来源
- Styling Session 完成后保存
- 摄像头 OOTD 捕捉后确认
- 用户手动添加

### 场景标签
用户在保存 Look 时可选场景（逛街 / 约会 / 通勤 / 运动 / 旅行 / 其他），不选则归入「全部」，不强制。

## 10. Style Identity（风格档案）

从用户行为自动推断，无需用户主动填写。

### 展示内容
- **风格关键词**：基于衣橱 style 标签 + Looks 历史推断（如 Clean / Minimal / Neutral Tone）
- **风格分布雷达图**：各风格维度占比可视化
- **趋势变化**：本月 vs 上月（如 Minimal ↑ / Street ↓）

### 更新逻辑
- 每次新增单品或保存 Look 后静默重算
- 用户不感知计算过程，只看结果

## 11. 色彩测试（Color Analysis）

### 入口
- Fashion Agent 快捷入口常驻
- 第一次进入搭配推荐时，在结果上方出现一次性引导卡片（可跳过，跳过后不再自动弹出）

### 测试形式
用户在自然光下拍一张面部照片，AI 判断肤色季型（春/夏/秋/冬型，细分如「浅夏 Soft Summer」），输出专属色卡与适合色系。

### 结果使用
- 存入用户 profile，参与搭配推荐排序
- 在推荐搭配中展示「适合你的色系」标注
- 用户可重测

## 12. Inspiration（灵感收藏）

用户收藏的穿搭灵感图，与自己的衣橱关联。

### 功能
- Tab 分类：全部 / 穿搭 / 色彩 / 搭配 / 收藏
- 用户从小红书/微博等分享进入 App 时可直接收藏
- 灵感图可触发「我的衣橱里有类似的吗？」检索（CLIP 相似单品查询）

## 13. 数据模型

### 单品（wardrobe_item）

| 字段 | 类型 | 说明 |
|---|---|---|
| `item_id` | string | UUID |
| `category` | string | 上装 / 下装 / 全身 / 外套 / 鞋履 / 配件 |
| `type` | string | 细分类型（见 data_format_v1.md） |
| `raw_type` | string | 原始识别名称 |
| `color` | array | 主色，1-3 个 |
| `style` | array | 风格标签 |
| `season` | array | 适穿季节 |
| `warmth` | string | 薄 / 中等 / 厚 / 不适用 / 无法判断 |
| `fit` | string | 修身 / 常规 / 宽松 / oversize / 不适用 / 无法判断 |
| `description` | string | 一句话描述 |
| `image_url` | string | 白底展示图路径 |
| `image_crop_url` | string | 原始裁剪图路径（CLIP 用） |
| `source` | string | real / screenshot / extracted / imported |
| `upload_time` | string | ISO 8601 |

### 用户 Profile（user_profile）

| 字段 | 类型 | 说明 |
|---|---|---|
| `user_id` | string | UUID |
| `photo_url` | string | 全身照路径 |
| `color_season` | string | 季型，空表示未测 |
| `color_tested_at` | string | 色彩测试时间 |
| `style_tags` | array | 主风格标签（自动计算） |

### OOTD 日志 / Look（look）

| 字段 | 类型 | 说明 |
|---|---|---|

## 14. 技术方案

### 图像生成
- 服务：`image2`（HTTP API，`image2_client.py`）
- 用途：Styling Session 试穿生成、穿搭照单品白底美化

### 衣橱检索
- 模型：**Marqo FashionSigLIP**（2025，Hugging Face 开源）— 在 100万+ 时尚商品图上 fine-tune，专门优化颜色/材质/类目/细节的图像相似度，比通用 CLIP 在 fashion 检索上高 57%
- 备选：SigLIP 2（Google，2025 年 2 月）— 通用底座，FashionSigLIP 不可用时的 fallback
- 场景：摄像头 OOTD 捕捉（按键触发）+ 后续版本 Inspiration「我的衣橱有类似的吗」
- 特征来源：优先 `image_crop_url`（原始上身 crop），无则用 `image_url`
- 索引：FAISS，本地构建，衣橱有变动时增量更新

### 天气
- GPS 定位 → 天气 API → 气温区间 → 映射 `warmth` 字段

### 存储
- 本地 SQLite（`wardrobe.db`），当前阶段无需云同步

## 15. 后续版本规划

| 功能 | 说明 |
|---|---|
| 穿搭频率提示 | 长期未穿单品提醒，减少衣橱通货膨胀 |
| 云同步 | 多设备衣橱数据同步 |
| 社交分享 | Lookbook 公开分享 |
| 灵感 → 衣橱关联 | 灵感图自动匹配相似单品并推荐补货 |
| 场景智能识别 | GPS + 日历自动推断场合，无需用户选择 |
| OOTD 镜子自动检测 | 摄像头 CV 检测到镜子场景自动提示，当前版本为硬件键触发 |
| Styling Session 细节调节 | 试穿时可调节：版型宽松度、拉链开合、下摆是否扎入、半扎等穿着方式 |
| 社交对象关联 | Look 可关联社交对象（见某人），系统记忆并在推荐时避免重复穿搭 |
| 个人冷热感知沉淀 | 麦克风捕捉用户语音反馈（「今天有点冷」），结合当时气温 + 单品 warmth 标签，逐步建立个人冷热感知 profile，让天气推荐更准 |
| OOTD 他人评价沉淀 | 麦克风捕捉他人对穿搭的评价（「今天好好看」），作为正向信号影响后续推荐权重 |
