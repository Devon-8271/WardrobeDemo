## 0. 产品定位

帮助用户**探索穿搭、表达风格**，而不只是整理衣橱。

| 竞品 | 定位 | 我们与之的差异 |
|---|---|---|
| 淘宝 AI 试穿 | 商家展示 SKU | 控制权在用户，支持换色/改版型/改穿法 |
| AICloset | 个人衣橱管理 + 穿搭记录 | 平铺推荐可进一步上身，原生中文，带数据飞轮 |

**我们的核心差异化能力：**
1. 自然语言控制试穿（fit / 穿法 / 颜色 / 风格）
2. 平铺搭配 → 上身试穿，两级 UX
3. 中文原生的个人审美档案（色季 + 体型 + 风格 + 温感）
4. 穿戴数据飞轮（越用越了解用户）
5. **AI 硬件原生**：摄像头 + 麦克风 + GPS 实现纯 App 做不到的被动感知——无需用户主动操作，数据自动积累

## 1. 用户档案（长期记忆）

用户档案是所有 AI 功能的基础上下文，存入记忆，每次推荐/评价/试穿调用时读取。

| 字段 | 说明 | 录入方式 |
|---|---|---|
| 身高 | 影响版型推荐 | 用户填写 |
| 体型 | 倒三角 / 梨形 / 沙漏 / 直筒 / 苹果 | AI 分析照片或用户选择 |
| 个人色季 | 冷冬 / 暖冬 / 冷夏 / 暖春 / 暖秋 | 测试问卷或 AI 分析肤色 |
| 风格偏好 | 街头 / 极简 / 复古 / 法式 等（多选） | Onboarding 选择 |
| 温感偏移 | 相对平均体感的冷热偏差（°C） | 推荐后主动反馈 / 麦克风 24/7 被动监听 |
| 禁忌 / 偏好 | 不喜欢的颜色、材质、场合 | 对话中积累 |

**Onboarding 流程（参考 AICloset，行业验证有效）：**
1. 上传至少 5 件衣物（强制，确保推荐有数据基础）
2. 选择 2-3 个风格标签
3. 完成色季小测试（可选，跳过后续可补）

## 2. 输入源

| 输入类型 | 具体内容 | 可达性说明 |
|---|---|---|
| 图片（主动） | 衣物照片（平铺/挂拍/上身）、穿搭照片、购物截图、灵感图 | 用户手机主动上传，用于新衣入库 |
| 摄像头（被动） | AI 硬件摄像头，320×240 / 20kb / 1fps 采样 | 用于行为检测 + 每日穿搭被动记录，见下方说明 |
| 麦克风（主动+被动） | AI 硬件麦克风，支持语音输入和 24/7 环境音采集 | 语音快速操作；被动捕捉温感抱怨和他人评价，见下方说明 |
| GPS 定位 | AI 硬件内置定位，自动获取当前城市 | 天气推荐无需用户手动输入城市 |
| 文字描述 | 场景需求、心情、风格方向、购物想法 | 对话输入 |
| 外部数据 | 实时天气（wttr.in，无需 key）、日期和季节 | 已接入 |
| 购买记录 | 淘宝/京东历史订单、邮件小票 | 未来方向，需用户授权 |
| 第三方平台 | 小红书收藏夹、品牌电商链接 | 未来方向 |

**摄像头输入的两级任务划分：**

```
新衣入库（一次性）
  └─ 需要高清图 → 引导用户用手机拍平铺照上传
     原因：识别面料/版型/细节需要分辨率保障

每日穿搭记录（被动持续）
  └─ 摄像头 20kb 可行 → 做闭集匹配
     任务：在已有衣橱（有限件数）里找最近似单品
     能力边界：
       ✓ 粗粒度：识别品类（上衣/裤子/外套）+ 主色方向
       ✗ 细粒度：区分相似单品（两件白T）、深色系（黑/深蓝/深灰）
     兜底：App 展示今日 log 供用户一键纠正，纠错数据反哺匹配精度
```

**技术实现：闭集检索（CLIP + FAISS Closed-Set Retrieval）**

> 核心范式转变：不是"识别世界上任意衣服"，而是"在用户已有衣橱中找最相似单品"。
> 难度下降一个量级，20kb 低清摄像头完全可行。

**新衣入库时（一次性，高清图）：**

```
1. GPT Vision → 结构化属性 JSON（type / color / fit / style / season / material）
2. OpenCLIP ViT-L/14 → flat-lay item_embedding → 存入本地 FAISS 向量索引
3. （冷启动增强，可选）如用户已完成数字人/形象建模：
     调用 try-on，将该单品生成到用户形象上
     → synthetic on-body image → 再生成 synthetic_onbody_embedding
     → 同一 item 存入第二条 embedding，用于提升初期识别率
```

**每日摄像头帧（持续，20kb 低清）：**

```
1. 对当前帧（或 SAM2 裁剪出的衣物区域）生成 CLIP embedding
2. FAISS Top-K 最近邻检索（不调 GPT，不做开放世界识别）
3. 返回 Top-3 候选单品给用户确认
4. 用户确认后：真实低清帧 embedding 沉淀为 real_onbody_embedding[]
5. 错误样本回流：用户纠正记录用于后续提升匹配准确率
```

**为什么用 retrieval 而非识别：**

| 维度 | 开放世界识别 | 闭集衣橱 Retrieval |
|---|---|---|
| 任务难度 | 极高（无限类别） | 低（20-200 件闭集） |
| 摄像头低清可行性 | 否 | 是 |
| 成本 | 每帧调 GPT Vision | 本地 CLIP 推理，接近零成本 |
| 延迟 | 秒级 | 毫秒级 |
| 准确率提升路径 | 无（外部模型固定） | 有（用户纠错 → real_onbody_embedding 积累） |

**每个 wardrobe item 的 embedding 数据结构（多来源）：**

```
item_embeddings（每件衣物维护三类来源）：
  flat_lay_embedding         ← 用户上传的高清平铺/挂拍/商品图（必选）
  synthetic_onbody_embedding ← try-on + 数字人生成的合成上身图（冷启动，可选）
  real_onbody_embeddings[]   ← 用户日常穿着被确认后的真实低清帧（持续积累）

检索时取最高相似度：
  item_score = max(
    sim(query, flat_lay_embedding),
    sim(query, synthetic_onbody_embedding),
    max(sim(query, e) for e in real_onbody_embeddings)
  )
```

> **synthetic on-body 的使用边界**：仅用于冷启动增强，不能替代真实样本。
> try-on 可能改变褶皱/光照，生成图可能美化版型，与真实硬件低清帧存在 domain gap。
> 最终准确率仍依赖：用户确认 → real_onbody_embedding 积累 → 错误样本回流。

**触发逻辑：**
- 摄像头持续低功耗采样，检测到「站在镜子前」行为 → 记录该时段帧
- CLIP 检索匹配衣橱 → 写入 outfit log
- 同一出行事件只记录一次，不重复写入

**麦克风输入的三种模式：**

```
① 主动语音操作（低摩擦替代打字）
  └─ 「帮我推荐今天约会的穿搭」→ 解析 TPO 场合，跳过菜单直接推荐
  └─ 「今天和小明吃饭」→ 解析人名 + 场合 → 写入社交场合记忆

② 被动温感监听（24/7 常驻，纯 App 做不到）
  └─ 检测到「好冷啊」「热死了」等关键词时自动触发：
     - 今日已有穿搭记录 → 直接更新 temp_offset（无需用户任何操作）
     - 今日无穿搭记录 → 询问用户穿了什么，建立上下文后再调整

③ 被动评价捕捉（纯 App 做不到）
  └─ 检测到他人对衣物的正面评价（「你这件好看」「这个颜色很适合你」）
     → 提示用户「要记录这条评价到 [单品名] 吗？」
     → 用户确认后写入单品评价记录
```

> **硬件原生优势**：摄像头 + 麦克风 + GPS 联动，纯 App 无法实现被动感知。所有被动采集均需用户二次确认才写入，不做无感后台录音。

**图像处理原则：**
- 新衣入库优先用手机高清照，保障识别质量
- 摄像头图像仅用于行为识别和穿搭记录，不用于建立衣物档案
- 不默认后台被动处理身体图像，行为检测结果不保留原始帧
- 处理完优先保存衣物属性，不长期保留原始全身照

## 3. 核心功能

### P0 — 基础 MVP（当前阶段）

**① 衣橱建模**
- 上传衣物照片 → AI 识别：品类 / 颜色 / 材质 / 季节 / 风格标签 / **版型（fit）**
- 识别结果逐件展示，用户可纠正品类、勾选/删减风格标签再入库
- 同款不同色必须分开为独立条目（不合并）
- 图片质量引导：拍摄前提示平铺白底、完整无遮挡
- 单品图美化（Beautify）：悬挂/手持/背景杂乱照片 → 生成干净平铺商品图
- source 字段区分真实入库（real）/ AI 生成（generated），推荐只读 real

**② 天气搭配推荐**
- 首页主动推送：启动时自动查天气（GPS 定位，无需输入城市），询问是否要看今日建议
- 按 TPO 场景分组：日常 / 工作 / 约会 / 旅行 / 运动 / 派对（支持数字选择或语音说「约会」）
- 按风格分组：用户偏好风格优先排序
- 推荐优先消耗衣橱内已有单品，不足时可提示补充
- 温感校准（双轨）：
  - 主动：推荐后询问冷/正好/热，更新 temp_offset
  - 被动：麦克风 24/7 监听，捕捉「好冷啊」→ 有穿搭记录时直接更新，无记录时先问用户穿了什么

**③ 用户档案采集**
- 身高、体型、风格偏好（Onboarding 填写）
- 个人色季（冷冬/暖春等）录入，推荐和换色时注入 prompt

---

### P1 — 差异化核心

**④ 两级穿搭展示**

```
Level 1 — 平铺搭配卡片（零 API 成本）
  ├─ 把衣橱单品图拼成 flat-lay
  ├─ 按 TPO 场景 + 风格分组展示
  └─ 用户快速浏览多套方案

          用户点击感兴趣的某套搭配
                    ↓

Level 2 — 上身试穿（用户有意向时才触发 API）
  ├─ 选择版型偏好：修身 / 正常 / 宽松 / oversize
  ├─ 选择穿法：默认 / 外套敞开 / 下摆扎入 / 半扎
  └─ 生成穿在用户自己身上的效果图
```

**⑤ 自然语言试穿控制**
- 换色：「把这件换成砖红色」→ 保留版型/印花，只改底色
- 改版型：「oversize 感穿」/ 「修身一点」
- 改穿法：「外套拉链打开穿」/ 「衬衫下摆扎进去」
- 风格迁移：「我想穿 Lemaire 那种风格」→ 生成同风格新搭配（需加"禁止照搬具体设计"约束）

**⑥ 精准试穿（Mask 路径）**
- 用户指定替换部位（只换上衣 / 只换裤子）
- 使用 Mask inpainting，只有目标区域进入 diffusion
- 头/肩/背景/其他部位完全不动，比例零变形
- 适用场景：修身款、对比例要求高的场合

---

### P2 — 数据飞轮

**⑦ 穿戴日历 + 使用数据**
- outfit_log 表：记录每天穿了什么单品、什么场合
- **写入方式（双轨）：**
  - 主动：用户手动确认今日穿搭
  - 被动：AI 硬件摄像头检测到照镜子行为 → 识别穿着 → 闭集匹配衣橱 → 自动写入，App 展示供用户一键纠正
- 统计每件单品穿戴次数和频率
- 如录入了购买价格，自动计算 cost-per-wear = 单价 ÷ 穿戴次数
- 衣橱排序支持：最常穿 / 最少穿 / 最久未穿 / 高穿戴成本 / 低穿戴成本

**⑧ 闲置推送 + 二手建议**
- 定期检测：超过 N 个月未穿的单品 → 推送"要不要整理一下？"
- 一键生成闲鱼商品描述 + 定价建议（参考原价和市场行情）
- 帮助用户决策：留着 / 出掉 / 改造

**⑨ 喜好分析**
- LLM 对比高频 vs 低频单品的属性（品牌/风格/场景/材质/价格区间）
- 归纳用户真实喜好，生成「你真正会穿的衣服长这样」报告
- 反哺推荐权重：偏好维度优先出现在推荐结果中

**⑩ 社交场合记忆**

User story：*作为用户，我在约见某人前，想看上次见他/她时自己穿了什么，避免重复；见面后我希望能快速记录这次见了谁、穿了什么。*

- outfit_log 扩展 `people` 字段：记录每次穿搭关联的社交对象（人名/标签）
- 见面前查询：「下次见小明，上次我穿了什么？」→ 检索 outfit log 中关联该人的记录
- 见面后快速记录：
  - 主动：App 内填写「今天和 [人名] [场合]」
  - 被动（硬件原生）：语音说「今天和小明吃饭」→ 解析写入，无需打开 App
- 单品维度查询：某件衣服曾经和谁穿过 → 辅助判断「这件还能再穿给他看吗」

**⑪ 他人评价记录**

User story：*作为用户，朋友夸了我某件衣服，我想把这条评价记下来；推荐时希望优先推那些收到过好评的单品。*

- 单品新增 `social_feedback` 字段：评价内容 + 评价人 + 日期
- 写入方式：
  - 主动：用户手动给单品添加评价
  - 被动（硬件原生）：麦克风检测到他人对衣物的正面评价 → 提示用户确认写入
- 推荐权重：高口碑单品在推荐排序中加分
- 衣橱视图：可按「最受好评」排序，快速找到社交场合的信心单品

---

### P3 — 长期方向

**⑬ 风格评价**
- 任意单品（衣橱内/照片/链接）丢进去
- 基于用户档案（色季/体型/风格偏好）给出评分（满分10分）+ 中文分析
- 可追问：「换成什么颜色更适合我？」→ 生成个性化色板

**⑭ 购买决策辅助**
- 用户截图/链接一件想买的衣服
- AI 判断：① 是否和已有衣橱容易搭配；② 是否符合用户色季和体型；③ 衣橱里是否已有类似的
- 输出：建议买 / 不建议买 + 原因

**⑮ 社区单品库 / 冷启动**
- 品牌单品搜索库（Nike/Supreme/Adidas 等），用户可直接加入衣橱
- 用户共享单品至社区库
- 购买记录导入：淘宝/京东历史订单 → 衣橱种子数据

## 4. 关键 Prompt 工程约束

> 以下是测试中发现的、必须在 prompt 层处理的问题，不处理会直接影响产品体验。

| 场景 | 必加约束 |
|---|---|
| 虚拟试穿（每次） | 保持人物面部/发型/肤色/肩宽/头肩比/姿态完全不变；不改变鞋子；服装保留原始商品特征，不允许 AI 自行美化 |
| 背景处理 | 将背景替换为干净的浅灰色摄影棚背景（默认）；或保留原背景（用户可选） |
| 风格迁移 | 参考图仅作风格参考，禁止复用任何具体服装设计，所有单品必须原创 |
| 颜色推荐 | 结合用户色季注入：「用户是冷冬型，优先推荐冷调颜色，避免暖土色系」 |
| 多轮变体 | 加"风格必须严格保持在 [标签] 范围内，不允许改变整体色调方向" |
| 衣橱建模 | 同款不同色必须分开为独立 item，不允许合并 |
| 图像生成（所有） | 负向词：no text, no logo, no watermark, no mirrored text, white background, flat lay, even lighting（由 prompt_builder 自动注入） |

**Embedding 架构说明（关键设计）：**

存两层 embedding，不合并：

```
item_embedding   — 单件衣物
  用于：闭集穿搭匹配 / 相似单品搜索 / 风格聚类

outfit_embedding — 整套穿搭组合
  用于：「推荐和这套感觉类似的」/ 穿搭风格分析 / P2 喜好分析
```

以上两层 embedding 支撑 P2 所有数据飞轮功能：风格聚类、「你真正会穿的衣服」报告、outfit compatibility 排序，均基于向量相似度而非纯 LLM prompt。

**模型选型建议（CLIP retrieval）：**

| 模型 | 用途 | 优先级 |
|---|---|---|
| OpenCLIP ViT-L/14 | 主力 retrieval baseline，精度更好，正式 feasibility test 用 | 高 |
| OpenCLIP ViT-B/32 | 快速跑通 pipeline，smoke test | 中（已有脚本） |
| SigLIP | embedding retrieval 对照实验，测试低清图鲁棒性 | 中 |

建议流程：先用 ViT-B/32 跑通 pipeline → 再用 ViT-L/14 做正式 feasibility test → 如 Top-3 仍不稳定，再补测 SigLIP。

> ViT-B/32 结果差不代表路线不行；正式可行性判断以 ViT-L/14 或 SigLIP 结果为准。

---

**Feasibility Test 方案（MVP 验收标准）：**

```
目标：验证低清摄像头帧能否通过 CLIP + FAISS 找回正确衣物。

测试流程：
1. 准备 10-20 件衣物高清图（各件外观有明显差异，覆盖不同颜色/品类/版型）
2. 用 OpenCLIP ViT-L/14 为每件衣物生成 flat-lay embedding，构建 FAISS IndexFlatIP
3. （可选）用 try-on 生成 synthetic on-body image，补充 synthetic_onbody_embedding
4. 以低清/模糊/暗光/局部裁剪图作为 query（模拟 320×240 摄像头帧）
5. 评估 Top-1 / Top-3 accuracy

验收标准：
  Top-1 ≥ 70%  → 可用于 MVP 自动识别，直接写入 outfit log
  Top-3 ≥ 90%  → 适合"自动识别 + 用户一键确认"模式（推荐 MVP 默认策略）
  Top-3 < 90%  → 默认展示 Top-3 供用户选择，不自动写入，等 real embedding 积累

已实现：demo/clip_retrieval_test.py（ViT-B/32 smoke test 版，已过滤生成输出）
待接入：ViT-L/14 正式测试 + synthetic on-body embedding 对比实验
```

## 7. 交付形式：Skill 架构

### 7.1 当前阶段
命令行 demo，验证核心逻辑。业务层（衣物识别 / 推荐 / 试穿 / 路由）已与 I/O 解耦，适合后续包装。

### 7.2 Skill 交付
最终产品以 **skill** 形式运行在 AI 硬件设备上（类似 Alexa Skill / iOS Shortcut），由硬件 skill runtime 调度。

包装时只需加一层薄适配器：

```
skill runtime
  ↓ 传入用户语音/文字
handle_input(text: str)       ← 新增，替换 main.py 的 input()/菜单
  → router.py（LLM 意图分类）
  → 对应 phase 模块处理
  → 返回结构化输出（文字 / 图片路径）给 runtime 渲染
```

现有 router.py / phase1-3 / db / image2_client **不需要改动**，只替换 I/O 层。

### 7.3 Skill to make skill（演进方向）

**当前**：router 从固定 skill 列表中选一个执行（dispatch）

**演进**：LLM 不只是选 skill，而是**动态编排多个 skill 串联执行**

```
用户：「出门前帮我看天气，推荐穿搭，再提醒我今天要见谁」

skill composer
  → [天气 skill] → [穿搭推荐 skill] → [社交场合记忆 skill]
  → 三个结果合并输出
```

用 LLM 来描述想要的功能 → 自动生成 skill 组合配置 → 注册为新 skill。这是 P3 阶段的架构方向，当前 router.py 是其雏形。
