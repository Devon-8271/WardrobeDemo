> **文档定位**：本文档是穿搭 Agent 所有 user story 的权威整合版本，统一来自原 [Omi分析/](../Omi分析/) 下的拆解稿（11 个老 US）、发散稿（时间维度与降级策略）、探索稿（可行性评估），并新增本次讨论沉淀的 6 个数字孪生 US（DT 组）。
>
> **路线背景（2026-05 转向）**：穿搭 Agent 由「Proactive 穿戴 + 文本/简单图像建议」升级为 **数字孪生路线** —— 脸部建模（face embedding，3-5 张自拍） + 可选体型（造型偏好，预设库） + 衣物上传 → 多条件合成。这一转向使「虚拟试穿」从原 PRD 中的"L4 独立能力，不作为核心 Story 驱动"升级为**支撑可视化主动建议的核心承载**。详见 [淘宝AI试穿拆解.md](淘宝AI试穿拆解.md) §4 与本文档第八节。
>
> **相关文档**：[../prds/digital-wardrobe-agent-skill-prd.md](../prds/digital-wardrobe-agent-skill-prd.md)（PRD 待按本文档重写） ｜ [淘宝AI试穿拆解.md](淘宝AI试穿拆解.md)（对标基线）
>
> **MVP 焦点（2026-05 起）**：17 个 US 是完整产品愿景，**MVP 阶段只做 4 个必做 + 1 个降级版**（US-LZ-02 / US-DT-01 / US-DT-02 / US-DT-03 + 简化版 US-FG-01）。详见 §九 MVP 优先级与边界 —— 任何与 §九 冲突的 US 内容以 §九 为准。

## 一、用户角色画像

### 角色 1：懒人穿搭者
> "我不想花时间想穿什么，但也不想太邋遢。帮我做决定就行。"

- 衣橱里衣服不少但每天只穿那几件；偶尔重要场合会焦虑
- 核心痛点：**选择疲劳 + 决策成本**
- 核心诉求：帮我省事、替我做决定、别让我动脑

### 角色 2：场合焦虑者
> "我最怕穿错场合。面试穿了 T 恤、朋友婚礼穿了运动鞋，这种事想起来就尴尬。"

- 对场合礼仪敏感但不一定知道规则；社交生活中有重要场合（客户/面试/约会/婚礼）
- 核心痛点：**穿错场合的尴尬风险**
- 核心诉求：别让我丢脸、在出门前阻止我犯错

### 角色 3：风格探索者
> "我想穿得更好看，但我不知道怎么搭。买了很多衣服但穿出来总是那几套。"

- 有提升穿搭的意愿但缺乏系统认知；会买新衣服但不知道怎么融入现有搭配
- 核心痛点：**对色彩/版型/风格没有明确概念**
- 核心诉求：教会我怎么搭、帮我在不买新衣服的情况下穿出新花样

### 角色 4：极简/理性消费者
> "我不需要 100 件衣服，我需要 30 件真正穿的。帮我算清楚每件衣服的使用率。"

- 在意衣橱效率和性价比；会定期清理但不确定该扔什么；买衣服前会考虑"使用率"
- 核心痛点：**衣橱效率不可见**
- 核心诉求：帮我去芜存菁、让每件衣服都有使用价值

### 角色 5：自我形象探索者 🆕（数字孪生路线新增）
> "我想看看'另一种可能的我'。穿这件衣服我会是什么样？10 年前的我穿现在的衣服又会是什么样？"

- 对自我形象有探索欲；愿意尝试但缺少低成本试错途径；有审美追求但常被现实约束（没拍照、衣服不在身边、场合不对）
- 核心痛点：**穿搭探索的现实摩擦力（必须真正穿上才能看到效果）**
- 核心诉求：让"想象中的自己"和"现实中的自己"之间有一座桥
- **关键说明**：这是数字孪生路线**才能成立**的角色 —— mask 路线下用户必须先穿上、再拍照，无法做"反事实探索"。

## 二、按角色组织的 17 个 User Story

---

### 角色一：懒人穿搭者（3 条）

---

#### US-LZ-01 自动记录每日穿搭

**作为** 不想手动记录穿搭的懒人
**我想要** Agent 自动记录我每天穿了什么
**以便于** 我不用动手就能知道自己穿了什么、穿了多久

**输入分层**：L1 ✓（穿搭颜色/风格粗识别）

**成功场景**：
- 每天出门 → 自动捕捉穿搭 → 静默归档，无需任何操作
- 同一件外套穿了多次 → Agent 说"这是你第 5 次穿这件黑色夹克了"
- 从不用打开 App 操作任何事

**失败场景**：
- 同一件衣服每次被识别为不同单品（衣橱里出现 5 件"蓝色衬衫"）
- 穿了但没捕捉到 → 漏记

**负面约束**：
- 必须零手动操作
- 捕获率不苛求 100%，但**去重必须准确**

---

#### US-LZ-02 早上帮我做穿搭决策 🔼

**作为** 早上不想纠结穿什么的懒人
**我想要** Agent 在起床后直接给我一个穿搭建议，并**让我看到化身穿上的样子**
**以便于** 我不用动脑、不纠结、不迟到

**输入分层**：L1 ✓ + L3 ✓（日历 + 天气）+ L4 ✓（衣柜建模 + 化身合成）

**成功场景**：
- 起床 + 日历有会议 → "今天有会，穿正式点。你那件白衬衫配深色西裤合适" + **化身穿这套的预览图**
- 起床 + 没安排 + 晴天 → "今天休闲就行，浅色 T 恤配牛仔裤" + **化身预览图**
- 用户对建议不满 → "再看一个"按钮 → 生成第 2 个备选化身图
- 衣柜未建模 → 退化为纯文字风格建议（不出图）

**失败场景**：
- 推荐了用户没有的衣服
- 化身体型/风格与用户感受偏差太大 → 用户不接受
- 用户已经穿好了才推荐 → 迟了

**负面约束**：
- 衣柜未建模时只推荐方向，**不推荐具体单品**
- 一天只推荐一次，不接受重复询问
- 起床后 30 分钟内推荐，超时不推
- 化身风格保持一致（不能这次是写实派、下次是卡通派）

**数字孪生路线下的能力增量**：
- 旧路线：仅能给文字建议（"穿那件白衬衫"）
- 新路线：可直接出图（化身穿这套站在符合场景的背景里），**说服力提升 1 个量级** —— 用户在出门决策窗口里第一次能"看到"今天的自己
- 这是淘宝 mask 路线**结构性做不到**的：mask 必须用户先有照片，主动触发完全废掉

---

#### US-LZ-03 告诉我该扔什么

**作为** 衣橱塞满但不知道哪些该扔的懒人
**我想要** Agent 告诉我哪些衣服很久没穿了
**以便于** 清理衣橱时不用自己翻记忆、直接看报告就能决定

**输入分层**：L1 ✓（US-LZ-01 的穿搭追踪数据积累）

**成功场景**：
- "这件灰色卫衣 2 个月没穿了，是忘了还是不喜欢？"
- 生成"活跃衣橱 vs 休眠衣橱"对比
- 换季时提醒："冬天过去了，这些厚外套可以收起来了"

**失败场景**：
- 季节原因没穿被误判为不需要
- 正式场合专用衣服（一年穿 2 次）被误判为休眠

**负面约束**：
- 休眠判定必须跨季节（至少 2 个月数据）
- 结合日历判断：正式服装可能只在特定场合穿
- 不做"你该买什么替代"的建议

### 角色二：场合焦虑者（3 条）

---

#### US-CJ-01 出门前场合穿搭检查 🔼

**作为** 害怕穿错场合的人
**我想要** Agent 在出门前判断我今天穿搭是否匹配场合，**并能让我看到在目标场合下的预览**
**以便于** 不在重要场合丢脸、不尴尬

**输入分层**：L1 ✓ + L2 ✓（音频："今天要见客户"）+ L3 ✓（日历）+ L4 ✓（场景化合成）

**成功场景**：
- 日历"14:00 客户提案" + 穿卫衣出门 → "今天见客户，卫衣可能太随意" + **化身在会议室场景下穿这身的预览** + 备选化身穿正装的预览
- 日历"朋友聚餐" + 穿西装 → "聚餐不用这么正式" + 备选化身预览
- 场合-穿搭已匹配 → 完全不打扰

**失败场景**：
- 去便利店也提醒 → 误判场合重要性
- 已出门才提醒 → 没时间换
- 化身在目标场景下渲染不真实 → 反而降低说服力
- 提醒语气像批评 → "你怎么穿成这样"

**负面约束**：
- 不确定时不提醒（宁缺毋滥）
- 一天最多 1 次场合提醒
- **公共场合禁止扬声器播报**（隐私红线）
- 提醒语气是"建议"不是"批评"
- 不重要场合不提醒（便利店/快递/熟朋友家）
- 提醒时机：**出门前 3-5 分钟**

**数字孪生路线下的能力增量**：
- 场景化预览：场景作为独立条件 → 同一化身可以渲染到「会议室坐着」「咖啡馆站着」「晚宴侧光」等不同场景
- 用户在出门前能直观对比"现在的穿搭" vs "建议的穿搭"两个化身图，决策摩擦降到最低
- 淘宝 mask 路线只能锁死原图场景，做不到这一项

---

#### US-CJ-02 重要人物穿搭不重复

**作为** 在意社交形象的人
**我想要** Agent 记住我见某个人时穿了什么
**以便于** 下次见同一个人时不穿重复的、给人新鲜感

**输入分层**：L1 ✓（穿搭粗识别）+ L3 ✓（日历参与人）+ L4 ✓（人脸识别 Fallback）

**成功场景**：
- 上次见客户王总穿蓝色衬衫 + 今天又是蓝色系 → "上次见王总穿的是蓝色衬衫，今天也是蓝色系，要换吗？"
- 见重要客户 → 提供上次见该客户的穿搭参考（可附化身重渲染图，见 US-CJ-03）
- 见不同人穿搭不同 → 静默积累社交穿搭档案

**失败场景**：
- 无法识别见面对象 → 不提醒
- 对不重要的人也提醒
- 提醒被理解为"监视社交关系"

**负面约束**：
- 人脸识别默认关闭，需用户授权
- 无法识别对象时不强行提醒
- 不重要对象不提醒（外卖/快递/熟人）
- 一天最多 1 次

---

#### US-CJ-03 见面前穿搭回顾 🔼

**作为** 要去见重要人物的人
**我想要** Agent 在我出门前回顾上次见这个人的穿搭，**并把历史穿搭重新渲染成统一画风**
**以便于** 心中有数、做出合适的穿搭选择

**输入分层**：L1 ✓（穿搭历史数据库）+ L3 ✓（日历参与人）+ L4 ✓（历史穿搭重渲染）

**成功场景**：
- 日历显示 30 分钟后见李总 → Agent 提供："你上次见李总穿的是深灰西装" + **化身穿那身的重渲染图**（即使原始拍摄不清楚）
- "这是你第三次见李总，前两次穿的分别是深蓝色和灰色系" + 3 张化身重渲染对比图
- 第一次见某人 → 不提供（无历史数据）

**失败场景**：
- 无历史数据时强行输出
- 重渲染历史穿搭时偏差太大 → 用户不认
- 提供信息过多 → 在出门前增加决策负担而非减轻

**负面约束**：
- 无历史数据时静默，不提示
- 只提供事实回顾，不给主观建议（"上次穿得好看"）
- 不作为主动推送，而是出门前 5 分钟轻量展示

**数字孪生路线下的能力增量**：
- 历史穿搭原本由 L1 低精度采集，画面质量参差 → 数字孪生让我们能用"化身 + 单品库"**重渲染**历史穿搭为统一画风的对比图
- 这把"历史回顾"从数据查询升级为视觉对比，决策辅助效率显著提升
- mask 路线绑定原图，没有重渲染的余地

### 角色三：风格探索者（3 条）

---

#### US-FG-01 基于实际穿搭的搭配发现 🔼

**作为** 想穿得更好但不知道怎么搭的人
**我想要** Agent 基于我实际穿过的衣服给我搭配建议，**并直接出图让我看到效果**
**以便于** 用现有的衣服穿出新组合、不需要买新的

**输入分层**：L1 ✓（穿搭历史数据库）+ L4 ✓（衣柜建模 + 化身合成）

**成功场景**：
- "你这件卫衣从来没和这条裤子搭过，试试？" + **化身穿这套搭配的预览图**
- "你总是深色上衣配深色裤子，要不要试试浅色裤子搭这件深色上衣？" + **化身预览图**
- 给出 3 个候选搭配 + 3 张化身图横向对比
- 基于已采集到的衣橱子集做搭配，不假设没见过的衣服

**失败场景**：
- 搭配建议离谱（完全不搭的两件硬凑）
- 建议的搭配用户早就试过但不喜欢
- 化身预览图与单品平铺图的视觉差异太大 → 用户不认
- 永远只推那几件的排列组合 → 无新意

**负面约束**：
- 只基于 Agent 确实见过的衣服做搭配
- 用户拒绝过的搭配不重复推荐
- 不确定时不推荐（避免离谱搭配拉低信任）

**数字孪生路线下的能力增量**：
- 旧路线：文字描述"卫衣 + 裤子配一下" → 用户必须想象效果
- 新路线：直接出搭配图 → 用户"看一眼就知道好不好" → **接受率天然高得多**
- 这才是「数字衣橱总管」的真正形态

---

#### US-FG-02 风格规律总结

**作为** 想知道自己穿衣风格是什么的人
**我想要** Agent 定期总结我的穿搭规律和偏好
**以便于** 了解自己的风格、在购物时有方向

**输入分层**：L1 ✓（穿搭追踪数据积累 — 至少 1 个月）

**成功场景**：
- "这个月你穿深色系占 70%，最常穿的风格是简约休闲"
- "你见客户时总是穿蓝色，蓝色是你的'正式色'吗？"
- "周一是你最正式的一天，周五最休闲"
- "你这季度开始尝试更多颜色了，不再是全黑"

**失败场景**：
- 数据太少时给出过度结论 → "你喜欢黑色"（只穿了一周）
- 总结太泛 → "你的风格是休闲"（什么都没说）

**负面约束**：
- 数据不足时给定性描述而非伪精确数字
- 至少需要 **1 个月数据**才能生成风格报告
- 不做"你应该尝试 X 风格"的建议（那是造型师的事）

---

#### US-FG-03 购物时判断新衣是否适合 ⬇️

**作为** 想买新衣服但怕买了不会穿的人
**我想要** Agent 在我考虑买一件衣服时告诉我它和现有衣橱搭不搭
**以便于** 减少冲动消费、只买真正会穿的衣服

**输入分层**：L1 ✓（已采集的衣橱子集）+ L4 ✓（拍新衣服照片，**不用化身试穿**）

**成功场景**：
- 在店里拍一件外套 → "这件外套和你常穿的 3 件内搭都配得上，值得考虑"
- "你已经有 2 件类似的蓝色衬衫了，确定还要吗？"
- "这件风格和你平时穿的差异很大 — 你觉得你会穿吗？"
- 给出"参考你过去 3 个月穿过的相似单品"，附旧穿搭真实历史图（不是化身）

**失败场景**：
- 只能和已采集的衣橱子集匹配 → 匹配范围有限
- 风格判断太主观 → 阻止用户买其实适合的东西

**负面约束**：
- 基于已采集衣橱子集做匹配，不声称覆盖完整衣橱
- 只做相容性判断，不做"这个品牌好"的推荐
- 不做电商导购链接
- **明确不依赖化身试穿** —— 详见下方"主动避开数字孪生的原因"

**主动避开数字孪生的原因（⬇️）**：
- 化身体型是**用户选定的造型偏好**，不等于真实体型 → "穿在化身上好看"≠"穿在我身上合身"
- 购物决策对"合身判断"敏感度高，化身-真身差异在这里会**误导消费**
- 我们的产品定位不是 GMV 工具，没有动机让用户"看起来合身就买"
- 替代路径：**用户过去穿过的相似单品历史图**作为决策辅助 —— 真实身体 + 真实穿搭，最有参考价值

### 角色四：极简/理性消费者（2 条）

#### US-JJ-01 衣橱使用率报告

**作为** 在意衣橱效率的极简主义者
**我想要** Agent 告诉我每件（已采集的）衣服的穿着频率
**以便于** 淘汰不穿的、优化衣橱结构

**输入分层**：L1 ✓（穿搭追踪数据 — 至少 2 个月跨季节数据）

**成功场景**：
- "这条裤子过去 3 个月只穿了 2 次，是你衣橱里使用率最低的单品"
- "这 5 件衣服贡献了你 80% 的日常穿搭"
- 生成"核心衣橱 vs 边缘衣橱"分布
- 换季时："这些夏装可以收起来了，过去 3 个月只穿过 1 次"

**失败场景**：
- 数据不够就下结论 → "这件衣服不穿"
- 季节性的正式服装被误判为不需要

**负面约束**：
- 必须跨季节数据才能下结论（至少 2 个月）
- 结合日历+场合判断：正式服装可能低频但必要
- 只做事实呈现，不做"你应该扔掉 X"

---

#### US-JJ-02 购物前的使用率预估

**作为** 不想买来压箱底的理性消费者
**我想要** Agent 在我买新衣服前预估它的使用率
**以便于** 判断这件衣服买回去会不会真正被穿

**输入分层**：L1 ✓（穿搭历史偏好数据）+ L4 ✓（拍新衣服照片）

**成功场景**：
- 拍一件亮色外套 → "你平时很少穿亮色，过去一个月亮色单品只出现过 1 次"
- "这件风格和你 80% 的穿搭一致，应该会经常穿"
- "这件和你已有的 3 件外套风格重叠，可能会出现'只穿其中 1 件'的情况"

**失败场景**：
- 基于历史数据过度推断 → 阻止用户尝试新风格
- 使用率预估不准 → 失去信任

**负面约束**：
- 预估用"可能/建议"语气，不做绝对判断
- 明确告知是基于历史数据的推断
- 不阻止用户买，只提供参考信息

### 角色五：自我形象探索者 🆕（数字孪生独家解锁，6 条）

> 这一组 6 个 US **完全依赖数字孪生路线** —— 在 mask 路线下要么做不到（受输入边界限制），要么做了也没意义（必须先有真实照片才能用，与"探索"诉求矛盾）。
>
> 共同特征：化身作为长期资产 + 单品库可任意组合 + 场景/姿态可参数化。

---

#### US-DT-01 离线穿搭演练 🆕

**作为** 通勤、午休等无镜子场景下想做穿搭决策的用户
**我想要** 在手机上直接预览"我穿这身的样子"
**以便于** 不依赖现场环境就能做穿搭决策

**输入分层**：L4 ✓（化身已建立 + 衣橱已采集）

**成功场景**：
- 午休时打开 App → 想试想穿明天那场会的衣服 → 化身穿出来给我看
- 通勤路上想"穿那件新外套配什么裤子" → 衣橱内任意组合，秒级化身预览
- 把化身 + 单品库带在手机里，不依赖镜子、不依赖回家试穿

**失败场景**：
- 生成延迟太长（>10 秒）→ 失去"演练"的轻量感
- 化身风格漂移 → 用户每次看到不一样的"我"，信任崩坏

**负面约束**：
- 化身风格保持一致，跨场景跨衣物不漂移
- 单品组合必须基于真实衣橱，不出现用户没有的衣服
- 默认离线缓存最近 N 件单品的合成结果，断网也能玩

**为什么 mask 路线做不到**：
mask 必须先有用户当下的真实照片才能编辑。"演练"的语义本身要求"不依赖现场拍照"，这与 mask 路线的根本前提冲突。

---

#### US-DT-02 反事实穿搭探索 🆕

**作为** 想试新风格但不敢冒险的用户
**我想要** 看到"如果我穿过 X" 的样子（X 可以是我没买过的衣服 / 没尝试过的风格）
**以便于** 在不消费的前提下尝试视觉风险

**输入分层**：L4 ✓（化身 + 候选单品图，可以是商品截图）

**成功场景**：
- 用户在小红书看到一件衣服截图 → 上传 → "你穿这件的样子"
- 用户没买过的"老钱风"风格 → 选定风格 prompt → 化身穿这种风格的搭配
- 用户对结果有判断 → 决定是否真的去买（这里不导向购物，但用户自然会去）

**失败场景**：
- 未明确标注"虚拟尝试" → 用户误以为是真实穿过的历史，污染穿搭数据库
- 把购物决策硬塞进来（"点这里购买同款"）→ 用户体感变低

**负面约束**：
- 必须明确标注"未尝试 / 虚拟探索" — 与真实穿搭历史严格隔离
- 不导购、不嵌购物链接（保持产品独立性，这是我们与淘宝最大的差异）
- 探索结果不进入 US-FG-02 的风格规律统计（避免污染）

**为什么 mask 路线做不到**：
mask 要求衣服已在原图里被穿着才能编辑。"反事实"的核心是"我从来没穿过的衣服" → mask 没有 mask 区域可参考，无法操作。

---

#### US-DT-03 场景化预览 🆕

**作为** 不确定一身造型在特定场合好不好的用户
**我想要** 看到这身造型在"会议室坐着 / 咖啡店站着 / 晚宴侧光"下的样子
**以便于** 在出门前评估场景适配性，不只是看正面立绘

**输入分层**：L4 ✓（化身 + 单品 + 场景预设库 5-10 个）

**成功场景**：
- 选定 1 套搭配 → 选场景：会议室 / 咖啡馆 / 晚宴 → 渲染 3 张不同场景化身图
- 用户发现"这身在会议室没问题，但在晚宴侧光下显得过于休闲" → 调整搭配
- 配合 US-CJ-01 自动触发：日历"晚宴" + 当前穿搭 → 直接渲染晚宴场景预览

**失败场景**：
- 场景库太少（只有 2-3 个）→ 用户感受不到价值
- 场景渲染的光照与化身不一致 → 化身像被 P 进去的
- 场景细节过多反而抢戏 → 用户看不清穿搭本身

**负面约束**：
- 场景预设库初期不超过 10 个（避免选择疲劳），但要覆盖主要场合大类
- 场景背景适度模糊，焦点保持在化身穿搭上
- 不允许用户自定义场景照片上传（V1 范围控制）

**为什么 mask 路线做不到**：
mask 锁死在原图场景。用户拍照在卫生间镜前，输出永远是卫生间镜前。换场景需要重新生成整张图。

---

#### US-DT-04 时间机器：风格演变可视化 🆕

**作为** 关注自己风格成长的用户
**我想要** 看到 3 个月前的我穿当时那身、对比现在的搭配
**以便于** 直观看到自己穿衣品味/审美的变化

**输入分层**：L1 ✓（历史穿搭数据库）+ L4 ✓（化身历史快照 + 重渲染）

**成功场景**：
- "你的风格演变" → 化身 3 个月前 vs 现在的对比图
- 用户看到自己从全黑 → 开始用彩色的视觉变化，建立"我在成长"的情感连接
- 季度风格年表：12 个月每月一张代表性化身图

**失败场景**：
- 化身没有历史快照 → 只能用当前化身渲染过去穿搭，对比维度只剩衣服
- 数据不足（用户用了不到 3 个月）→ 没东西可对比
- 把"过去"渲染得过于美化 → 用户体感失真

**负面约束**：
- 化身**必须有版本快照**（每次脸部资产更新时保存旧版本）
- 至少需要 3 个月数据才生成时间机器报告
- 不做"你以前更好看"或"你现在更成熟"等主观评价

**为什么 mask 路线做不到**：
mask 没有"化身"概念，每张试穿图独立存在，无法回顾历史。这个 US 是数字孪生**版本化资产**的天然衍生品。

---

#### US-DT-05 衣橱全景生成 🆕

**作为** 不知道自己衣橱里有多少种搭配可能的用户
**我想要** Agent 把我衣橱里所有"上 × 下"组合自动渲染出来
**以便于** 看到一个"我从未发现的自己"

**输入分层**：L1 ✓（衣橱已采集）+ L4 ✓（化身 + 批量合成）

**成功场景**：
- 用户主动触发"看我的所有搭配可能" → Agent 渲染 20-30 个高价值组合
- 用户在网格视图里浏览，发现"原来我衣橱里能穿出这么多种"
- Agent 标注"这 5 个你已经穿过、这 15 个还没试过"

**失败场景**：
- 组合数爆炸（10 件上 × 10 件下 = 100 张）→ 用户没耐心看完
- 没做筛选，把明显不搭的组合也渲染出来 → 整体观感拉低
- 全量渲染消耗推理预算太大 → 成本不可持续

**负面约束**：
- 必须做"搭配价值"筛选 —— 只渲染 Top 20-30 个高价值组合
- 筛选规则透明：可以告诉用户"我跳过了这些组合，因为颜色冲突 / 风格冲突"
- 频率：用户主动触发为主，月度自动生成 1 次为辅
- 推理预算控制：单次全景批量任务限定上限

**为什么 mask 路线做不到**：
mask 需要每件组合都有用户对应的真实照片，这显然不可能。批量预渲染是数字孪生路线的天然能力。

---

#### US-DT-06 化身代我分享 🆕

**作为** 想问朋友意见但不想发自拍的用户
**我想要** 把化身穿搭图发给朋友，而不是发真实自拍
**以便于** 在保留隐私的同时获得社交反馈

**输入分层**：L4 ✓（化身 + 单品合成图）

**成功场景**：
- 用户在家素颜 → 试穿明天会议穿搭 → 直接把化身图发给朋友（不用先化妆拍照）
- 用户在外不方便拍照 → 用化身图问伴侣意见
- 用户分享到社交网络（朋友圈/小红书）→ 主动隐去真实自拍，化身做代理

**失败场景**：
- 化身辨识度太高 → 等于发了"接近自拍"，隐私价值打折
- 化身辨识度太低 → 朋友看不出"是你" → 反馈失去意义
- 默认带水印过于显眼 → 用户不愿分享

**负面约束**：
- 化身分享时**默认带"AI 合成"标识水印**，可关闭
- 不允许用化身代替本人做身份验证类操作
- 用户可控分享化身的"像我程度"（用滑块调整辨识度）

**为什么 mask 路线做不到**：
mask 的"试穿图"本质上还是用户的真实自拍 + 换衣服，隐私属性等同于自拍。化身的隐私价值依赖于"它不是我的真实照片"这一前提，这是数字孪生独有的属性。

## 五、与传统数字衣橱 App 的本质差异

```
传统数字衣橱 App 的 User Story：
  作为用户，我想要把所有衣服拍照上传
  以便于 App 告诉我怎么搭配

淘宝 AI 试穿的 User Story：
  作为用户，我想要在购物时看到衣服穿在我身上的样子
  以便于决定买不买

我们的 User Story（数字孪生路线）：
  作为用户，我想要 Agent 自己学会我的穿搭习惯并维护我的数字孪生
  以便于在我需要看见/决定/比较时，Agent 都能主动提供视觉化的、跨时空的穿搭可能性

本质差异：
  传统衣橱：用户主动 → App 被动响应 → 衣橱是静态资产库
  淘宝试穿：用户购物时主动点击 → App 编辑照片 → 试穿是 GMV 工具
  我们：    Agent 主动观察 → 主动建议 → 试穿是数字孪生的可视化输出

  传统：用户每次都要打开 App
  淘宝：用户必须在购物意图下才用
  我们：Agent 在用户没意识到的时候就开始工作
```

## 六、降级策略（数据不足时）

每个 US 都需要明确"数据不足时怎么办" —— 这是产品稳定性的根基（数字孪生路线下扩展自原发散稿）：

| 场景 | ✓ 仍可输出 | ✗ 不输出 |
|---|---|---|
| 用户刚开始用（Day 0-3）| 化身建立引导、L1 穿搭追踪开始（静默）| 任何穿搭建议、可视化、风格总结 |
| 化身已建立但衣橱数据不足（Week 1）| 纯文本风格建议、天气提醒、场合检查（不出图）| 化身可视化建议、搭配发现 |
| 衣橱数据足够但化身风格漂移 | 退化为纯文字版本所有 US | 任何化身可视化输出 |
| 跨季节数据不足（< 2 个月）| 当季穿搭追踪、本月风格定性总结 | 衣橱使用率、该扔什么 |
| 化身快照不足（< 3 个月）| 当前所有可视化 US | 时间机器 US-DT-04 |
| 用户拒绝过的搭配 | 其他搭配建议 | 重复推荐被拒搭配 |
| 主动触发置信度不足 | 主动建议静默 | 强行推送 |
| 公共场合 | 微震 / 私密语音 | 扬声器播报 |

**核心原则**：数据不足时的降级**不是"做差一点"，而是"换一种形态"** —— 把可视化退回文字、把主动退回被动、把"该扔什么"退回"今天穿了什么"。降级路径必须明确，避免出现"垃圾输出污染用户信任"的情况。

## 七、明确不做的事（吸收自发散稿 + 数字孪生路线下的新增红线）

```
不做：
  ├── 推荐购买什么衣服（购物推荐需要电商集成，与产品独立性冲突）
  ├── 单次深度造型（婚礼伴娘装等需要专业造型师视角，非我们强项）
  ├── 社交分享互评（社交功能不做，但化身分享 US-DT-06 是个例外，那是隐私工具）
  ├── 衣物护理管理（干洗/修补，不是核心场景）
  ├── 衣橱总价值评估（需要完整衣橱 + 价格数据，做不准）
  ├── 心情穿搭关联（用户需求弱、技术可靠性差，参考发散探索稿评估）
  ├── 约会场景特化（需要"对方偏好"数据，违反隐私边界）
  ├── 化身的极端体型变化（如"瘦 30 斤"假设）→ 心理健康红线
  ├── 化身变脸为他人（明星 / 朋友）→ 法律和道德红线
  ├── 儿童化身建立 → 红线
  └── 化身用于身份验证类操作 → 安全红线
```

## 八、产品决策注记（决策追溯历史）

> **当前状态**：5 个决策点已在 2026-05 全部收敛，权威结论见 [§九.4](#94-5-个产品决策点已收敛2026-05)。本节保留作为**决策追溯历史**，记录每个决策点曾经的候选方案与最终选择背后的权衡。
>
> 任何 PRD 阶段的执行以 §九.4 为准；本节仅供"为什么不选另一个方案"的回溯。

### 决策 1：化身风格 —— 真实派 vs 风格化派

- 真实派（照片级）：用户接受上限高，但 uncanny valley 风险大
- 风格化派（接近真实但故意保留绘画感）：长期容忍度高，工程更宽松
- **最终选择（2026-05）**：**真实派**（产品底线）。理由：现有模型已能生成人脸真实的图像；MVP 必须对标淘宝的真实感体感；uncanny valley 不作为降级理由。

### 决策 2：体型表达 —— 「我是」vs「我喜欢的视觉」vs 自动学习

- 「我的真实体型」：心理负担重，可能与真实身体冲突
- 「我的造型偏好」（显瘦 / 显挺拔 / 显宽松）：心理触感轻，但隐含主观判断
- 自动从图片学习 + 用户校准：贴近真实但保留控制权
- **最终选择（2026-05）**：**三条路径并行** —— 自动从自拍学习（主路径）+ 用户可输入身高/体重 + 用户可选预设参考体型。

### 决策 3：化身是否为用户的人格资产

- 是 → 化身有昵称、个性化界面、用户有强心理所有权（参考游戏角色）
- 否 → 化身只是工具，简洁呈现
- **最终选择（2026-05）**：**不做虚拟形象** —— 化身即"数字化的用户本人"，与真实派风格保持一致。明确切割"造型伙伴 / 游戏角色"等人格化方向，不作为后期演进方向。

### 决策 4：化身的入口 —— 穿戴还是手机 App

- 穿戴主入口：与 Proactive Agent 一致，但可视化受限
- 手机 App 主入口：可视化好，但摩擦力高
- **最终选择（2026-05）**：**手机 App 主入口**；穿戴硬件仅作为图像 / 语音输入源，不做化身呈现。这是产品形态的关键澄清。

### 决策 5：化身的产品红线

- 不允许极端体型变化（瘦 30 斤等假设）→ 心理健康
- 不允许变脸为他人 → 法律 / 道德
- 不允许儿童化身 → 红线
- 不允许化身用于身份验证类操作 → 安全
- **最终选择（2026-05）**：**全部 4 条写入红线**，从 MVP 第一天就生效，所有路径必须在 PRD 中体现执行机制。

## 九、MVP 优先级与边界

> **本节定位**：把上面 17 个 US 的"完整愿景"切成 V1 可落地的 MVP 范围。前八节是产品方向的天花板，本节是产品落地的地板。MVP 阶段任何与本节冲突的 US 调整都以本节为准。
>
> **MVP 核心问题**：在**数据有限 + 硬件未发布**的前提下，证明「数字孪生路线 + 主动建议」相比「淘宝 mask 路线」的差异化是**真实可演示的**。

### 9.1 MVP 设计原则

四条硬约束决定了 MVP 的 US 取舍：

1. **避开依赖长期数据的 US** —— 自动记录、跨季节使用率、时间机器全部后置。MVP 不能要求用户"先用 1 个月再开始享受价值"。
2. **避开依赖穿戴硬件的 US** —— L1 端侧自动追踪、出门自动检测、自动场合识别在 MVP 阶段全部砍掉，因为穿戴形态本身还没上线。
3. **保留数字孪生独家能力** —— 这是 MVP 与淘宝拉开差距的唯一筹码。如果不展示试穿 / 反事实 / 场景预览，等于做了个功能更弱的淘宝。
4. **保留主动建议雏形** —— 即使主动触发降级为"用户手动开 App，Agent 给一张今日建议"，依然要比淘宝纯被动的产品形态有体感差异。

### 9.2 17 个 US 的 MVP 优先级

| US | 状态 | 理由 |
|---|---|---|
| US-LZ-01 自动记录 | ❌ **Backlog** | 依赖穿戴硬件 + 长期 L1 数据 |
| US-LZ-02 早上决策 🔼 | ✅ **V1 必做** | 数字孪生路线的主入口；手动触发版本即可上线 |

### 9.3 MVP 最小可上线闭环

从用户首次启动到第一次价值兑现的链路：

```
[首次启动]
    ↓
[化身建立] 3-5 张自拍 + 选造型偏好 ── 约 5 分钟
    ↓
[衣物上传] 5-10 件衣物（拍照 / 商品截图皆可）── 约 10 分钟
    ↓
[首次价值兑现] 「早上一张」化身穿这身的预览图（US-LZ-02）
    ↓
[反馈收集] 用户接受 / 不接受 → 数据飞轮起点
    ↓
[二次使用入口]
    ├── "再看一个" → 离线演练模式（US-DT-01）
    └── "我看到小红书一件衣服" → 上传截图 → 化身试穿（US-DT-02）
```

**关键指标**：从首次使用到第一次"出图被接受" ≈ **15-20 分钟**。这个时长决定了 MVP 留存的天花板。

### 9.4 5 个产品决策点（已收敛，2026-05）

§八列出的 5 个待决策点已在 MVP 阶段全部收敛，**PRD 阶段直接按下表执行**：

| 决策项 | 已定方向 | 关键说明 |
|---|---|---|
| **化身风格** | ✅ **真实派**（产品底线） | 现有模型已能生成人脸真实的图像，对标淘宝的真实感体感。这是 MVP 的产品底线 —— 任何形式的风格化 / 卡通化都不接受，即使工程更难也要做真实派。需密切监控 uncanny valley 反馈，但**不作为降级理由**。 |
| **体型表达** | ✅ **三条路径并行**：① 自动从上传的自拍中学习 ② 用户可手动输入身高 / 体重 ③ 用户可从预设体型库中选"参考体型" | 自动学习是主路径（对标淘宝的"看到就是真实自己"体感）；手动输入和参考体型作为校准 / 调整入口。三条路径**不互斥**，用户可任选 / 组合使用。 |
| **化身入口** | ✅ **手机 App 主入口；穿戴硬件仅负责图像 / 语音输入** | 这是关键的形态澄清 —— 穿戴硬件 = 数据采集器（不做呈现），手机 App = 化身建立 / 试穿 / 浏览 / 反馈的中心。化身的所有交互都在手机 App 内完成。 |
| **化身人格** | ✅ **不做虚拟形象，化身即"数字化的用户本人"** | 不起独立昵称、不做游戏角色形态、不做拟人化交互。化身的本质是"用户在数字世界的真实自己"，与真实派的风格选择一致。与"造型伙伴"等人格化方向**明确切割**，不作为后期演进方向。 |
| **产品红线** | ✅ **4 条硬红线，从 MVP 第一天就生效** | (1) 不允许极端体型变化假设（"瘦 30 斤"等）→ 心理健康；(2) 不允许变脸为他人 → 法律 / 道德；(3) 不允许儿童化身 → 红线；(4) 不允许化身用于身份验证类操作 → 安全。所有路径必须在 PRD 中体现这 4 条的执行机制。 |

**5 条决策的内在一致性**：

```
化身风格：真实派   ──┐
                       ├── 化身 = "数字化的用户本人"，不是"用户的卡通分身"
化身人格：不做虚拟  ──┘
                       
体型表达：自动 + 校准 ─┐
                          ├── 化身的所有属性都尽量贴近真实用户，但用户保留控制权
入口：手机 App       ─┘

产品红线：4 条       ─── 真实派化身的安全 / 伦理边界
```

所有 5 条决策共同指向一个产品定位：**「数字化的用户本人」是化身的核心隐喻**。这与"卡通化身 / 游戏化分身 / 风格化形象"等其他可能性**明确互斥**。

### 9.5 MVP 明确不做的事

为避免范围蠕变，提前固定这些 V1 边界：

```
不做：
├── 任何穿戴硬件相关的自动捕捉（L1 端侧追踪、自动场合识别）
├── 长期数据驱动的能力（风格规律、使用率、时间机器、该扔什么）
├── 见面对象识别 / 人脸识别（隐私 + 数据稀缺，V3 才考虑）
├── 衣橱全景批量预渲染（算力预算紧，V2）
├── 化身分享 / 社交（V2）
├── 购物决策辅助带化身试穿（避免化身误导消费，V2 重新设计）
├── 真实店铺 / 商品 / 电商导购集成（保持产品独立性）
└── 主动推送（V1 仅"手动触发的主动建议"，不做后台推送）
```

### 9.6 MVP → V2 路径

MVP 验证完后的下一步演进顺序（数据驱动，不预设具体时间）：

1. **V1.5（数据沉淀 1-2 个月后）**：开放 US-FG-02 风格规律 + US-FG-01 完整版搭配发现 —— 这两个需要 1 个月数据，是 V1 的自然延展。
2. **V2（穿戴硬件预备时）**：开放 US-CJ-01 场合检查 + US-DT-05 衣橱全景 + US-DT-06 化身分享 —— 三个增强类能力。
3. **V3（穿戴硬件正式上线）**：开放 US-LZ-01 自动记录 + US-CJ-02/03 社交穿搭 + US-DT-04 时间机器 —— 依赖长期数据 + 硬件的完整能力。

### 9.7 V1 的成功标准（拍脑袋暂定，待数据验证）

- **量化**：化身建立完成率 > 60%；首次"早上一张"接受率 > 40%；7 日留存 > 25%
- **质化**：用户至少能说出一个我们能做、淘宝做不到的事（如"我让它试了张小红书上的衣服" / "我看了化身在我没拍过的姿态下穿这身的样子"）
