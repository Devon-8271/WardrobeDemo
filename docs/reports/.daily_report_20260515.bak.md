### 6. 开会讨论点

**讨论点 1 — AI Look 保存时机（⚠️ v1.1 暂不做）**

结论：OOTD 保存统一存单品拼图卡，不弹选项。AI Look 事后从 Looks 页触发，留 v1.2。

理由：生成阻塞保存流程；弹 3 选项违背零摩擦；双层 timeline 需先验证用户行为和生成质量。

**讨论点 2 — 用户全身照上传时机与无照 fallback**

背景：试穿效果图依赖用户全身照，但首次强制上传违背零摩擦原则，且部分用户可能只想管理衣橱、看搭配建议，不一定接受上传自己的人像。

方案 A（推荐）：**软提示 + 延迟门控**
- 首页顶部常驻软上传 bar（提示但不阻断），可关
- 进入"我的形象"页 `/profile` 主动上传/更换
- 触发需要照片的功能（生成试穿效果图）时再弹轻提示：「试穿效果图需要先上传全身照，去上传 / 跳过看文字搭配」
- 无照场景下：推荐卡片只展示单品拼图（每套 4 件白底图拼成 2×2），caption + 风格标签照旧

方案 B：首次强制上传（之前考虑过的方案）
- 进首页就弹全屏 modal，不传不能用
- 优点：试穿是核心卖点，强制能保证体验完整
- 缺点：违背零摩擦，劝退只想浏览/管理衣橱的用户

**讨论焦点**：
- 无照用户的产品价值是否成立？（看推荐 + 管衣橱 + 看日志，没试穿）
- 拼图替代试穿图的视觉表达够不够？
- 触发上传的时机（首次进首页 / 点"生成"按钮 / 永远软提示）

待会议确认。

---

**讨论点 3 — Tomorrow Planning Hub 推送是否含可滑动 outfit 卡片**

背景：4-5 套效果图已预生成，推送时图已就绪，复用零额外成本。

方案 A：Hub 推送直接展示横滑 outfit 卡片，用户在 Hub 层浏览，点套进 Fashion Agent 调整，少一跳。

方案 B：推送纯文字 CTA，点进 Fashion Agent 再看图，职责分离更清晰。

待确认：Hub 层是否支持横滑卡片交互，是否符合 AI Hub 整体交互规范。

```
  → image2_client.generate() (拼图 + PIL 裁切)
  → 返回 {action, payload, message} → 前端按 action 渲染卡片

  → POST /upload/wardrobe-item

  → 立即返回 (存原图)

  → 衣橱页 8s 轮询 → badge 消失 → 图自动替换
```

**遇到的坑**：

| 问题 | 原因 | 修复 |
|---|---|---|
| 前端天气栏溢出整个页面 | `wttr.in/?format=%t+%C` 偶发返回完整 HTML 页 | 改用 JSON API `wttr.in/?format=j1` + header 加 `overflow:hidden` |
| 推荐图全部 404 | `_to_url()` fallback 取 `p.name`，丢了 `grid/` 子目录 | 区分绝对/相对路径，相对路径前直接加 `/` |

| 单品上传后未美化 | Beautify 代码加入前上传的单品没机会触发 | 新增 `POST /api/wardrobe/{id}/beautify` + 弹窗内"重新美化"按钮 |
| 详情图被裁切 | `.modal-img` 用了 `object-fit: cover` | 改 `contain` + `max-height: 420px` |

**Beautify 异步设计要点**：image2 一次调用约 1-5 分钟（含连接重试），不能阻塞 HTTP。用 FastAPI `BackgroundTasks` 立即返回 → 后台任务跑 image2 → 完成后更新 DB；前端通过 `_beautifying` 内存集合返回 `beautifying:true` 标志，未完成时显示 badge，8 秒轮询，完成自动替换。image2 不可达时跳过 beautify 但仍入库，不影响主流程。

### 9. "我的形象"模块 + Web 端 user_profile bug 修复

**起因**：原 `/upload/photo` 端点存在两个 bug：

| Bug | 现象 | 根因 |
|---|---|---|
| 上传必崩 | KeyError: 'upload_time' | 端点向 `db.upsert_user_profile()` 传字典时漏掉 `upload_time` 必填项 |
| 上传抹数据 | 旧 CLI 设的 `height / body_type / skin_tone / personal_color` 全被清空 | `upsert_user_profile` 是全字段 UPDATE，端点传 `""` 覆盖了已有值 |

之所以 DB 里仍能看到完整 profile，是因为之前的数据是 CLI 时代 `phase2_tryon.setup_user_profile()` 写入的；Web 端从未真正成功写过。`user_photos/` 下有两张文件但 DB 没记录，就是 bug 留下的现场。

**修复**：
- 新增 `db.update_user_photo(user_id, photo_url, upload_time)` — 窄更新，只动这两个字段，保留其他
- `/upload/photo` 改用窄更新

### 10. 视觉识别 prompt 收敛 — 解决"背景陪衬被误识别"

**问题**：早期 prompt 写"识别照片中的所有衣服…不允许省略任何单品"，跟 web 上传"一次一件主体"的实际场景反了。实测一张商场粉色背心展示图被识别成 **6 件**（1 件主体 + 5 件背景货架上的鞋）。

**修复**：重写 `_VISION_PROMPT`（`phase1_wardrobe.py`），加入三条主体判断规则：
1. 占面积最大、位于中心、对焦清晰
2. 平铺/挂起/手持的（明显被刻意拍摄）
3. 同款不同色组合算多件

同步明确"必须忽略"清单：背景货架、衣架陪衬、模糊/遮挡半数以上、镜子反射、模特身上的（除非整张就是模特展示）。并加一句兜底：**宁可漏识别，不要把背景当主体；没有主体就返回空数组**。

**效果验证**：同一张粉色背心图，识别从 6 件 → 1 件，背景的红色高跟鞋/棕色平底鞋/黑色平底鞋/米色平底鞋/白色拖鞋全部正确忽略。

**备选方案（暂未实施）**：客户端裁切——上传前让用户框选主体区域再发送。如果改 prompt 后仍有边缘 case 漏切，再上裁切组件。

### 12. 推荐质量优化（隔离改动，不动 templates / DB）

把推荐链路从「random shuffle 取 N」升级为「打分排序 + 日级缓存」。三件事，全在 `outfit_recommender.py`：

**12.1 每日推荐缓存**

```python
_CACHE: dict[(user_id, date_iso, temp_bucket)] -> list[outfit]
```

- key 含温度分桶（cold ≤5 / cool ≤15 / warm ≤25 / hot），天气变化才重算
- 缓存里 item_id 失效（用户删了单品）→ `_cache_still_valid` 自动重算
- 空结果不写缓存（衣橱回填后能立即生效）
- `refresh=True` 强制跳过
- 衣橱变动钩子：`app.py` 的 `/confirm/wardrobe-item` 和 `DELETE /api/wardrobe/{id}` 调 `outfit_recommender.clear_cache()`

效果：用户刷新首页不再换组（之前 random.shuffle 每次都不同），符合 PRD「默认停在第一套」。

**12.2 风格档案加权排序**

读 `user_profile.style_tags`（`style_identity.py` 回写的），衰减赋权 `1.0 / 0.8 / 0.6 / 0.4 / 0.2`。候选 combo 得分 = `Σ style_w[s] for s in combo_styles`。无 style_tags 时 cold start，退化为随机。

**12.3 最近穿过的组合降权**

读最近 7 天 `db.get_looks()` 的 `item_ids` 集合，对候选 combo 算重叠率，重叠 ≥50% 时扣分（`-overlap`）。避免周一推完周二再推同一套。

打分后加微量 `random.random() * 0.1` 扰动，同分组合不会永远定死。

**测试**：`test_recommender.py` 从 4 个加到 **8/8 通过**：

| 新增测试 | 覆盖 |
|---|---|
| `test_daily_cache` | 同 user/日期/温度段两次调用结果相同；refresh=True 可重算 |
| `test_cache_invalidates_on_wardrobe_change` | mock 衣橱缩减后缓存自动失效 |
| `test_style_weighting` | 偏好「法式」时 10/10 首套含法式 tag |
| `test_recent_look_penalty` | 最近穿过的 a1+b1 组合 0/10 排首套 |
