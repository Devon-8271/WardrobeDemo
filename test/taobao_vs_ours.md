# 竞品对比：淘宝 AI 试穿 / AICloset vs 我们的方案

> 基于实际试用两款产品后的体验记录（2026年5月）。目标不是全面超越，而是找到我们真正能赢的点。

---

## AICloset 产品介绍

> 实际下载试用版本，以下为功能全景梳理。

### 基本信息
- 定位：个人数字衣橱管理 + AI 穿搭助手
- 支持语言：英文为主（AI 回复默认英文，中文用户体验较差）
- 平台：iOS / Android

### 核心功能模块

**① 衣橱（Wardrobe）**
- 多种录入方式：相册上传 / 拍照 / 网页链接 / 品牌库搜索 / 邮件小票自动导入 / 电商平台直连（SHEIN、GAP、ZARA 等）
- 上传后自动识别：Category（分层，如 Outerwear > Sweat Zip-up）、Color、Season、Brand、Occasion、Pattern、Purchase price、Material
- Beautify 功能：对悬挂/手持/背景杂乱的照片做清理，输出干净平铺商品图（效果可用）
- 单品可手动编辑所有字段，支持打 Label / 写 Notes
- 衣橱排序维度丰富：最常穿 / 最少穿 / 最近穿 / 最久未穿 / 高穿戴成本 / 低穿戴成本 / 按颜色 / 按类别 / 按添加日 / 按购买日 / 价格高低

**② 穿搭（Outfits）**
- 推荐输出形式：平铺单品拼图（flat-lay collage），不是上身效果图
- 按 TPO 场景分组：日常 / 学校 / 工作 / 旅行 / 派对 / 约会 等
- 按风格分组：休闲 / 经典 / 街头 / 极简 / 现代 等
- 优先使用衣橱内已有单品，不足时推荐衣橱外商品
- 每组场景/风格下有刷新按钮，可一键重新生成该场景的搭配方案

**③ 首页**
- 根据明天天气自动推送搭配建议（主动 push，不需要用户触发）
- 天气信息显示：日期 / 城市 / 最低最高气温
- 热门功能快捷入口：添加物品 / 创建穿搭 / 日历 / 美图改造 / 风格统计

**④ AI 功能（首页快捷入口）**
- **穿搭建议**：基于用户档案（色季 + 体型 + 风格偏好）给出穿搭方向
- **造型聊天**：对话式穿搭咨询，可追问，支持对结果继续提问
- **找到适合我的颜色**：基于个人色彩季型（冷冬/暖春/冷夏/暖秋等）推荐颜色，并生成专属色板
- **找到我的版型**：分析体型（倒三角/梨形/沙漏等），给出适合版型建议
- **评价风格**：任意单品（衣橱内/照片/搜索结果）丢进去，基于用户档案打分（满分10分）并给出个性化评价和改进建议
- **虚拟试穿**：可将衣物试穿到用户身上，实测版型/身材比例真实感较好

**⑤ 用户档案（长期记忆）**
- Onboarding 强制入库 5 件 + 选择风格偏好，确保推荐有足够数据
- 存储：个人色季（如"冷冬 Winter Cool"）/ 体型 / 风格偏好
- 所有 AI 功能均读取此档案，做个性化输出

**⑥ 穿搭日历 + 数据统计**
- 记录每天穿了什么（outfit log）
- 统计每件单品的穿戴次数和频率
- 如录入了价格，自动计算 cost-per-wear（单次穿戴成本）= 单价 ÷ 穿戴次数
- 排序维度"最久未穿"可定期筛出闲置单品，推动二手处理
- 风格统计：分析衣橱整体风格分布

**⑦ 社区库**
- 品牌单品库：搜 Nike AF1、Supreme 等可直接出结果，用户可将单品加入衣橱
- 用户可将自己的衣橱单品共享至社区库供他人使用
- 形成网络效应：社区越大，库越丰富，录入摩擦越低

## 一、vs 淘宝 AI 试穿

### 对比总览

| 问题维度 | 淘宝 AI 试穿 | 我们的方案 |
|---|---|---|
| 颜色选择 | ❌ 部分商品不支持换色，控制权在商家 | ✅ 自然语言换色，任意颜色均可指定 |
| 尺码/版型宽松度 | ❌ 锁定商品默认版型，无 fit 控制 | ✅ Prompt 注入 fit 偏好（修身/宽松/oversize） |
| 穿法样式 | ❌ 只有"展示状态"，外套不能拉开穿 | ✅ 自然语言描述穿法，支持 styling 变体 |
| 人物比例保真 | ❌ VTON warping 易导致头肩比失调 | ✅ Mask inpainting 只动衣服区域，比例严格锁定 |
| 版型可读性 | ❌ 只有图，看不出具体版型信息 | ✅ 结构化版型字段 + 多角度生成 + 文字标注 |

### 逐条拆解

**1. 颜色选择**

淘宝：部分商品支持换色，部分不支持，控制权在商家。

我们：B2 换色已验证（Step 7），自然语言指定颜色，版型/印花不变。产品层做颜色 picker 自动注入 prompt。
```
把这件上衣的颜色改成墨绿色，保留所有版型、细节和印花不变。
```

---

**2. 尺码 / 版型宽松度**

淘宝：只能穿商品默认版型，想要 oversize 但生成板正，无法控制。

我们：Prompt 注入 fit 偏好，产品层做选项 **修身 / 正常 / 宽松 / oversize**，自动拼接。
```
以 oversize 方式穿着，衣服宽松落肩，不贴身，袖子有余量
```

---

**3. 穿法样式**

淘宝：只有商品展示状态，外套永远扣好，无法模拟敞开穿。

我们：自然语言描述穿法，Step 6 多轮修改已验证可执行。
```
外套拉链完全打开，随意敞开穿，里面露出内搭
衬衫下摆全部扎进裤子里，腰线收紧
```

---

**4. 人物比例保真**

淘宝：专用 VTON warping 时肩线易被拉宽/压缩，修身款头肩比异常。

我们：优先走 Mask inpainting（D1），只有衣服区域进 diffusion，头/肩/背景完全不动（Step 10 验证零变形）。

---

**5. 版型可读性**

淘宝：只有正面图，面料/腰线/袖长等版型细节无法判断。

我们：① 结构化版型字段；② 多角度生成（正面 + 四分之三侧面）；③ 版型标注叠加。

---

**核心差异：** 淘宝 try-on 是**商家的展示工具**，控制权在商家；我们做的是**用户的探索工具**，控制权在用户。

### AICloset 的关键弱点

**弱点 1：推荐结果是平铺拼图，不是上身试穿**

AICloset 的穿搭推荐把单品图拼在一起做成 flat-lay 卡片，不是穿在人身上的效果。用户看到的是"这几件可以搭"，但看不到"穿在我身上什么样"。这是其架构天花板——没有生成式模型支撑，无法跨越到上身这一步。

**弱点 2：AI 输出默认英文，不适合中国用户**

实测：评价风格、颜色建议、穿搭分析全部英文输出，对中国用户体验极差。"冷冬/暖春"这套季节色彩理论在小红书极度流行，但 AICloset 的英文回复完全切断了这个场景的传播性。

**弱点 3：无自然语言控制**

所有操作是点击/选择，无法表达：
- 「oversize 感穿，外套敞开」
- 「把这件换成砖红色试试」
- 「我想穿 Lemaire 那种风格，重新设计一套」

这是 VTON + 规则架构的本质限制，不是功能缺失，是技术路线问题。

---

### 我们的产品设计方向

**① 两级 UX：平铺 → 上身**

```
Level 1 — 平铺搭配卡片（快，零成本）
  ├─ 按 TPO 场景 + 风格分组
  ├─ 优先用衣橱内单品拼图
  └─ 用户快速浏览多套方案

            用户点击感兴趣的搭配
                     ↓

Level 2 — 上身试穿（只在用户有意向时触发 API）
  ├─ 选择 fit 偏好（修身 / oversize）
  ├─ 选择穿法变体（敞开 / 扎入）
  └─ 生成穿在用户自己身上的效果图
```

AICloset 在 Level 1 就停了。我们多走一步，差异化恰好在最关键的决策节点——"这套穿在我身上好不好看"。API 成本只在用户真正感兴趣时产生，可控。

**② 用户审美 DNA：原生中文的个人档案**

AICloset 已验证"个人色季 + 体型 + 风格偏好"作为长期记忆的价值——所有 AI 功能从这里读取，推荐和评价都是个性化的。

我们要做，且要做成中文原生：
- **个人色季**（冷冬/暖冬/冷夏/暖春/暖秋）— 小红书热门话题，自带传播属性
- **体型**（倒三角/梨形/沙漏/直筒）— 影响版型推荐
- **风格偏好**（已有） + **温感偏好**（已实现，带反馈学习）

全部存入 user_profile，推荐/换色/风格迁移调用时注入 prompt。

**③ 穿戴日历 + 数据飞轮**

AICloset 的穿戴日历设计很有价值，我们可以做并深化：

- 记录每天穿了什么（outfit log 表）→ 统计穿戴频率
- 录入价格后自动计算 cost-per-wear
- **长期未穿 → 推荐出二手**：国内对接闲鱼，可一键生成商品描述 + 定价建议（这个功能本身有传播性）
- **高频单品分析**：LLM 读取高频 vs 低频单品的属性差异（品牌/风格/场景/材质/价格区间），归纳用户真实喜好，反哺推荐逻辑

这个数据层让推荐从"风格标签匹配"升级到"基于你的真实穿衣行为"，越用越准。

### 冷启动策略

1. **Onboarding 强制入库 5 件** — 行业验证有效，直接采用
2. **风格偏好 + 色季选择** — 入库后完成，推荐时用作权重
3. **Level 2 不依赖衣橱** — 风格探索（C3）第一次进来就能玩，不需要完整衣橱
4. **购买记录导入** — 淘宝/京东历史订单、小红书收藏解析为种子数据（未来方向）

---

**④ AI 硬件原生：被动感知层**

这是淘宝和 AICloset 在架构上无法追赶的一层——它们是纯 App，没有常驻硬件，所有数据必须靠用户主动输入。我们的硬件提供三路被动感知：

| 硬件能力 | 淘宝 AI 试穿 | AICloset | 我们 |
|---|---|---|---|
| 城市/定位 | ❌ 无 | ❌ 手动输入 | ✅ GPS 自动获取，推荐无感知 |
| 温感反馈 | ❌ 无 | ❌ 无 | ✅ 麦克风 24/7 监听「好冷啊」→ 自动更新偏移 |
| 穿搭记录 | ❌ 无 | ❌ 手动记录 | ✅ 摄像头被动识别今日穿着 → 自动写入 outfit log |
| 社交场合记录 | ❌ 无 | ❌ 无 | ✅ 语音「今天和小明吃饭」→ 写入，无需打开 App |
| 他人评价捕捉 | ❌ 无 | ❌ 无 | ✅ 麦克风听到「你这件好看」→ 提示记录到单品 |
| 出行前穿搭提醒 | ❌ 无 | ❌ 无 | ✅ 摄像头检测照镜子行为 → 推送试穿/确认邀请 |

**核心优势**：纯 App 的数据飞轮需要用户持续主动操作才能转动；我们的数据飞轮是**被动积累**的——用户只需要正常生活，系统自动变得越来越了解他。这个差距随时间拉大，不可追赶。

## 三、竞品定位对比总结

| 维度 | 淘宝 AI 试穿 | AICloset | 我们的方案（AI 硬件） |
|---|---|---|---|
| 核心定位 | 商家展示 SKU | 个人衣橱管理 + 穿搭记录 | 穿搭探索 + 风格创作 + 被动感知 |
| 推荐输出 | 商品页试穿图 | 平铺单品拼图 | 平铺 → 可上身（两级） |
| 控制权 | 商家 | 用户（点选） | 用户（自然语言 + 语音） |
| 语言 | 中文 | 英文为主 | 中文原生 |
| 用户档案 | 无 | 色季 + 体型 + 风格（英文） | 色季 + 体型 + 风格 + 温感（中文） |
| 数据写入方式 | 无 | 全部手动 | 主动 + 被动双轨（摄像头/麦克风自动积累） |
| 数据飞轮 | 无 | 穿戴日历 + cost-per-wear | 穿戴日历 + 闲置推送 + 喜好分析 + 社交记忆 |
| 定位 | 无 | 手动输入城市 | GPS 自动获取 |
| 温感学习 | 无 | 无 | 24/7 麦克风被动捕捉，无需主动反馈 |
| 冷启动 | 不需要（商品库） | 社区库 + 强制 5 件入库 | 需要设计（学 AICloset） |
| 护城河 | 淘宝流量 | 社区内容库 + 网络效应 | 硬件被动感知层 + 生成式控制 + 中文原生 |
| 我们能超越的点 | 颜色/版型/穿法控制，比例保真 | 平铺→上身，自然语言，中文档案，被动数据飞轮 | — |
