# GPT-image-2 能力测试指南

> 目标：不是测好不好看，是测底座能不能用。先跑基础，别上来测高级风格。

## Step 1：Text-only 单品生成 ✅

**目的：** 测模型对服装描述的基础理解。

**Prompt：**
```
生成一件深咖色长款双排扣大衣，正面平铺，白底电商图。
```

**观察项：**
- [x] 颜色是否正确（深咖色）→ 准确，深棕咖色
- [x] 双排扣有没有体现 → 有，6 颗扣清晰可见
- [x] 长款比例是否对 → 对，过膝长款
- [x] 版型是否合理 → 合理，略宽松廓形，肩线自然

**结论：** 全部通过。Text-only 对纯色、结构清晰的单品描述还原度高。

## Step 2：Text-only 整套穿搭生成 ✅

**目的：** 测基础搭配能力和单品完整性。

**Prompt：**
```
生成一套 cleanfit 女性全身穿搭：米白针织衫、浅蓝阔腿牛仔裤、棕色乐福鞋，干净高级，白底 lookbook。
```

**观察项：**
- [x] 三件单品是否都出现（针织衫 / 牛仔裤 / 乐福鞋）→ 全部出现，且图中有标注
- [x] 搭配是否协调 → 高度协调，整体 cleanfit 感准确
- [x] 人体比例是否正常 → 正常，面部自然，无崩坏
- [x] AI 味重不重 → 极低，几乎可以直接当真实 lookbook 用

**额外注意：** 模型自行添加了深棕色手提包，prompt 未要求。说明模型会主动补全穿搭，需注意后续产品中是否要约束这个行为。

**结论：** 全部通过。整套穿搭还原度和真实感超出预期。

## Step 3：单品图 → 衣橱建模 ⚠️ 基本可用

**目的：** 测最核心的产品能力——能不能从照片提取结构化属性。

**输入：** Fano 品牌蓝白格纹无袖款，挂衣架拍摄，背景杂乱

**实际输出：**
```json
{
  "category": "不规则无袖上衣",
  "color": ["蓝色", "白色"],
  "material": "棉质格纹梭织面料",
  "silhouette": "宽松不规则剪裁，层叠垂坠版型",
  "style_tags": ["法式", "日系", "cleanfit", "设计感", "度假风", "轻文艺"],
  "season": ["春季", "夏季"],
  "occasion": ["日常出街", "咖啡店", "brunch", "旅行", "约会"]
}
```

**观察项：**
- [x] 品类有没有搞错 → ⚠️ 存疑：图中衣长接近连衣裙，叫"上衣"可能不准；挂着拍难判断
- [x] 颜色是否准确 → ✅ 蓝白格纹，准确
- [x] 有没有瞎编材质或版型 → ✅ 棉质梭织合理，版型描述符合图示
- [x] 结果是否可以直接写入数据库 → ✅ 结构完整，可直接写入

**已知问题：**
- `style_tags` 中"cleanfit"用于格纹不规则款语义偏怪，cleanfit 通常指纯色极简
- 拍摄条件差（挂架、背景杂乱）时 category 准确率下降，建议平铺或白底拍摄

**失败案例记录：腰部叠穿裙片（layering overskirt / 屁帘）**
- 输入：深蓝米黄格纹叠穿裙片，揉成一团斜放桌上拍摄
- 模型输出：`"category": "格纹吊带连衣裙"` — 完全错误
- 失败原因：① 拍摄姿态差，结构不清晰；② 品类冷门，模型强行映射到最近的已知品类"连衣裙"，把系带误认成肩带
- 颜色/材质仍然准确（深蓝+米黄，法兰绒质感）
- 结论：**冷门品类 + 非标准拍摄 = 必错**，品类越非标越需要用户手动确认

**失败案例记录：同款不同色合并入库**
- 输入：两件版型完全相同、颜色不同的衣服（如白色和黑色同款 T 恤）
- 模型默认行为：将两件合并为一个 item，只记录其中一种颜色，另一件丢失
- 失败原因：裸 prompt 下模型倾向于"去重"，把同款视为一件
- 修复 prompt：在识别指令中明确加入 `"请严格按照照片中的单品逐件识别，不允许合并同款不同颜色的衣物。即使版型相同，颜色不同也必须分开为独立 item"`
- 修复后验证：两件分别作为独立 item 返回，颜色均正确 ✅
- 代码层联动：识别结果改为返回 JSON 数组，`upload_clothing` 改为逐件确认入库

**重点记录：** 识别准确率与两个变量强相关——拍摄质量（平铺白底 > 挂拍 > 随手堆放）+ 品类常见度（基础款 > 设计款 > 冷门叠穿件）。

## Step 4：单品平铺图 → 上身效果 ✅ 超出预期

**目的：** 虚拟试穿的低难度版，先测衣服保真度。

**输入：** 白色 #80 美式球衣 + 灰色阔腿牛仔短裤，两件一起平铺在床上拍摄（实际为双件输入，比计划难度高）

**Prompt：**
```
保留这件衣服的颜色、版型和主要细节，生成一张真人上身效果图。
```

**观察项：**
- [x] 颜色是否保持 → ✅ 白色球衣+红色印花、灰色牛仔短裤，颜色准确
- [x] 版型是否保持 → ✅ 阔腿短裤比例对，球衣 oversize 感保留
- [x] 细节（图案/印花）有没有被改掉 → ✅ #80 数字、ASIF CALIE 印花基本保留
- [x] 上身效果是否自然 → ✅ 镜拍姿态自然，人体比例正常

**已知问题：**
- 球衣文字**镜像反转**（模型学了镜拍风格但没识别出文字不能反）
- 自动补全了白袜 + Adidas 板鞋，原图未提供
- 本次实为**双件平铺**输入，比 Step 4 原计划难度高，效果仍然好，说明多图理解能力强

**结论：** 复杂图案款（印花+多件）上身还原度超出预期，主要风险是含文字的图案会镜像失真。

## Step 5：人 + 单品 → 虚拟试穿 ✅ 效果强

**目的：** 核心难点，也是产品能做的上限。

**输入：**
- 用户全身照：洛杉矶路口街拍，黄昏光线，人物佩戴粉色棒球帽（脸部被遮挡）
- 参考衣物：Step 4 测试的 #80 球衣 + 灰色阔腿短裤（平铺图）

**Prompt：**
```
保持人物脸、身材、姿态不变，只把上衣替换成参考图中的衣服。
```

**观察项：**
- [x] 人脸有没有变化（漂移）→ ⚠️ 无法验证：帽沿遮住脸，本次测试有盲区
- [x] 衣服款式颜色是否保持 → ✅ #80 球衣 + 灰色短裤正确应用，颜色准确
- [x] 是否只换了目标衣物，其他没动 → ✅ 帽子/发型/姿态/背景全部保留；⚠️ 鞋子被自动替换为白袜+Adidas
- [x] 身体比例是否崩 → ✅ 腿部暴露更多但比例自然，无崩坏
- [x] 背景是否乱变 → ✅ 背景完全一致，光影/棕榈树/路灯无变化

**结论：** 整体效果非常好，几乎可用于产品展示。两个固定问题：① 脸部遮挡时无法验证人脸保真；② 鞋子会被模型自动替换，建议在 prompt 中加"不要改变鞋子"。

---

**补充测试：双人图指定操作 + 多轮修改**

- 输入：两人合照（左：金发女生穿灰色西装外套；右：京剧妆持花束）
- 操作：指定只换左边人的上衣为 #80 球衣
- 第一轮结果：球衣正确应用，但下摆被塞进裤子，衣长偏短
- 第二轮修正："不把衣服塞进去" → 放出来了，长度改善

**新增关键结论：**
- ✅ **正脸人脸保真验证通过** — 换衣后人脸无漂移，发色/五官/表情一致
- ✅ **双人图可指定操作** — 只改左边，右边京剧妆+花束完全未动
- ✅ **多轮修改有效** — 一句话指令可以修正上一轮的具体问题
- ⚠️ **衣长由参考图决定** — 上传的单品图下摆被遮挡 → 生成的衣服偏短；单品参考图必须完整无遮挡

**产品要求更新：** 用户上传单品图时必须提示"请确保衣物完整可见，不要遮挡任何部分"。

**建议测试顺序：** 基础款（白T）→ 正式款（衬衫）→ 复杂款（图案 / oversize）

## Step 6：多轮修改 ✅ 可用，但有系统性偏差

**目的：** 测能不能做"穿搭对话"，一致性能撑几轮。

**实际测试流程（基于真实衣橱沉淀，不是预设 prompt）：**

第一轮：
```
生成一套去散步的穿搭。
```
→ 生成：#80 jersey + 浅蓝阔腿裤 + 粉帽 + 黑托特包。用户反馈：太不搭，jersey 不适合散步。

第二轮：
```
不要这个包，不要 jersey。
```
→ 生成：米白针织衫 + 浅蓝阔腿裤 + 粉帽。休闲感对了，但 **粉帽自动降饱和变成莫兰迪粉**。

第三轮（更新 prompt 后）：
```
保持人物、场景和整体构图自然。但服装单品必须严格保留原始商品特征，不允许 AI 自行美化或改色。
目标：真实穿搭试穿效果，而不是时尚杂志二次设计。现在给我生成一套遛狗的穿搭。
```
→ 生成：白色长袖印花上衣 + 深灰牛仔五分裤 + 粉帽 + 白袜 + Adidas + **模型自行添加了一只白色泰迪犬**。上衣图案出现变形错误。

**观察项：**
- [x] 人物是否在三轮内保持一致 → ✅ 发型/肤色/体型稳定
- [x] 每轮是否只改了该改的部分 → ✅ 基本只改指定内容
- [x] 风格有没有崩掉 → ✅ 整体保持户外休闲感
- [x] 有没有越改越崩 → ✅ 没有，每轮都在收敛

**关键发现：**

**① 全局调色一致化（Aesthetic Harmonization）**
模型会自动把高饱和单品（亮粉帽）降调为莫兰迪/胶片色，以匹配场景整体氛围（黄昏/暖光/韩系 cleanfit）。
- 原因：diffusion 模型默认优化"审美统一"而非"商品保真"
- 修复：prompt 加 `"cap color must exactly match the reference image"` / `"保持帽子原始高饱和亮粉色，不允许降低饱和度"`
- 产品结论：**AI 会把东西变好看，但不一定尊重商品原色** — 电商/虚拟试衣场景需要 fidelity，不是 aesthetic reinterpretation

**② 场景语义驱动补全**
"遛狗"→ 模型主动生成了一只狗。与之前补鞋、补包、补配件是同一逻辑：
- 模型补全"场景合理性"，不只响应用户明确要求
- 产品上需要决策：是允许这种创意补全，还是严格约束只输出用户衣橱内的单品

**③ 图案保真仍是弱点**
即使加了保真 prompt，上衣印花图案仍出现变形/重绘。含文字/复杂图案的单品保真度差，是 diffusion 模型的固有局限。

**④ 创意生成是保守的"最小编辑"**
要求"换一件风格一致的短袖，不一定要衣橱里的"→ 模型直接把上一轮的长袖袖子截短，而不是创作一件新的短袖。
- 帽子替换正常执行（粉色→深蓝棒球帽）
- 模型的创意偏好：**最小改动 > 真实创作**，不会主动"设计"新单品
- 产品结论：用这个模型做"穿搭提案"时，如果想要真正新的单品推荐，需要在 prompt 里明确禁止复用现有图像，或者走 text-only 生成通道而不是 edit 通道

## Step 7：B2 — 单品换色 ✅

**目的：** 测局部颜色替换，衣橱"没有这个颜色但想试试"的场景。

**输入：** #80 白色球衣 + 灰色短裤平铺图

**Prompt：**
```
把这件上衣的颜色改成墨绿色，保留所有版型、细节和印花不变。
```

**观察项：**
- [x] 颜色是否改对 → ✅ 白→军绿/橄榄绿，在绿色系内；⚠️ 偏军绿而非正墨绿，颜色语义有轻微偏移
- [x] 其他部分（版型/印花/结构）是否保持不变 → ✅ #80 红色数字、asifcalie、AIC 文字全部保留
- [x] 改色后面料质感是否自然 → ✅ mesh 光泽感保留，不像贴图
- [x] 印花/logo 颜色是否跟着变 → ✅ 红色印花未跟着变色，保持独立

**结论：** 单品换色能力强，中英文颜色词效果无明显差异。

**产品价值：** 用户可以用同一件衣服试不同配色，不需要买多件。

## Step 9：C3 — 风格图 + 人物 → 风格迁移（测试中）

**目的：** 测"我想穿这个风格"场景——用户发一张 inspo 图，让模型把自己穿进去。

**测试轮 1：风格差距小**

- 风格参考图：黑色超宽松 graphic tee + 深色宽腿工装裤 + 墨绿帽（archive/Y2K 街头）
- 用户输入图：La Palma 户外，黑色 Stussy tee + 黑色工装裤（本身已经是同风格）
- 结果：上衣换成了深色复古印花 tee，裤子几乎没动，背景/人物完整保留

**现象分析：**
- 模型发生了"**风格锚定（style anchoring）**"：把宽松工装裤识别为风格核心 token，由于原图裤子已高度吻合，认为不需要替换
- 风格差距越小 → 迁移 delta 越小 → 结果越接近原图微调

**测试轮 2：风格差距大**

- 风格参考图：深色无袖上衣 + 灰色不规则中裙 + 过膝靴 + 月牙包（Lemaire / Our Legacy 设计师品牌气质，feminine-minimal）
- 用户输入图：同一场景镜前自拍

**第一轮（无约束 prompt）：**
→ 模型直接把参考图的具体衣服照搬到用户身上，literal copy，不可用

**第二轮（加 "禁止复用具体设计" prompt）：**
```
参考图片的整体风格氛围，但禁止复用参考图中的任何具体服装设计。
请重新设计一套全新的穿搭。
允许：相同风格语言 / 相同审美方向 / 相同品牌气质
但所有单品必须是新的原创设计。
```
→ 生成：深灰扭结短袖 + 宽腿西装裤 + 尖头靴 + 黑色斜挎包，风格方向正确

**观察项：**
- [x] 人物身份是否保留 → ✅ 金发/墨镜保留
- [x] 是否复制参考图人脸 → ✅ 没有
- [x] 风格差距大时迁移是否成功 → ✅ 方向正确，深色结构感 feminine-minimal 氛围对
- [x] 单品是否全新 → ✅ 扭结上衣是原创设计，未照搬

**新发现：背景环境 bleed**
背景从用户原始场景变成了参考图的店铺环境（棕色瓷砖地板、玻璃搁板），模型把参考图的 environment token 一起迁移了进来。说明 style anchoring 不只影响服装，也影响背景。
→ 修复方向：prompt 中加 "保持用户原始照片的背景环境不变"

**结论：** 风格迁移可用，但需要 prompt 工程。默认行为是 literal copy，需要显式禁止才能触发真正的风格迁移；背景 bleed 是额外副作用需要约束。

**这是产品差异化的核心能力之一**：从"穿我衣橱里的"升级到"穿我想要的风格"。

## Step 10：D1 — Mask 精准替换 ✅

**目的：** 测局部 inpainting，实现"只换上衣，裤子/背景/人脸一分钱不动"。

**输入：**
- 用户全身照（La Palma 户外，黑色 Stussy tee + 黑色工装裤）
- 上衣区域 mask

**Prompt：**
```
只替换 mask 区域的上衣为：白色宽松短袖T恤，其他所有区域严格保持不变。
```

**观察项：**
- [x] Mask 外的区域是否完全没动 → ✅ 背景/裤子/鞋/阴影全部保留
- [x] Mask 内边缘过渡是否自然 → ✅ 衣领和袖口边缘过渡自然
- [x] 替换的衣服与周围光影是否匹配 → ✅ 白色 tee 与户外暖光匹配，无违和感
- [x] 姿态/体型在换衣后是否变形 → ✅ 完全没有变形

**结论：** 所有测试中精准度最高的操作方式。Mask 是实现"只换目标部位"最可靠的手段，副作用最小，是虚拟试穿产品的最优技术路径。代价是需要生成 mask 图，产品上需要考虑 mask 怎么让用户提供或自动生成（如用分割模型自动框出上衣区域）。

## 结论表（测完填）

| 阶段 | 是否可用 | 主要失败点 |
|---|---|---|
| Text 单品生成 (A1) | ✅ | 颜色/结构/比例全部准确 |
| Text 整套穿搭 (A2) | ✅ | 单品完整，真实感强；模型会自行补全未指定配件 |
| 衣橱建模 | ⚠️ 基本可用 | 颜色/材质准，非标准拍摄时 category 易偏，style_tags 有语义噪音 |
| 平铺→上身 (B1) | ✅ | 颜色/版型/印花还原好；含文字图案会镜像反转；会自动补全配件 |
| 虚拟试穿 (C1) | ✅ | 正脸保真验证通过；可指定双人图中某人操作；衣长取决于参考图完整度 |
| 多轮修改 (E组) | ✅ | 人物一致性稳；调色一致化会改变单品原色；场景词会触发模型自行补全道具 |
| 单品换色 (B2) | ✅ | 颜色/版型/印花保真好；中英文颜色词效果无明显差异 |
| 整套上身 (C2) | ✅ | Step 4/5 已覆盖，两件同时应用正常，腰线衔接自然 |
| 风格迁移 (C3) | ⚠️ 需 prompt 工程 | 默认 literal copy；加禁止复用后方向正确；背景会被参考图环境 bleed |
| Mask精准替换 (D1) | ✅ | 精准度最高，副作用最小；需要 mask 图，产品需考虑自动分割方案 |

## 额外发现：多参考图风格推断 + 变体生成

**测试：** 给 3 张不同的风格参考图（dark archive street / dark feminine / grunge），让模型推断用户偏好并生成穿搭

**第一轮结果（✅）：** 黑色工装外套 + 宽腿工装裤 + 链条腰带 + 胸前 harness，与参考气质高度吻合；背景替换为干净灰色混凝土墙成功

**第二轮（"再给我一个不一样的"）（⚠️ 跑偏）：** 变成深棕色镂空针织 + 棕色工装裤，向 mocha 秋日户外漂移，出了 dark archive 风格边界

**根本原因：** "不一样"的指令权重高于"忠实参考风格"，模型为了和上一轮区分，开始探索相邻风格区域，出了边界

**与 Step 6 发现的对比：**
- Step 6：模型换衣时"最小编辑"，不够大胆
- 本测试：模型求变时反而走太远，风格漂移

**两种失败模式共存：模型在"保守"和"过度发散"之间没有稳定的中间状态**

**修复 prompt：**
> "再生成一套不同单品组合，但风格必须严格保持在参考图的 dark archive street 范围内，不允许改变整体色调方向"

## 待解决产品体验问题

### 试穿结果背景太杂乱

**问题：** 用户上传的全身照背景是真实场景（卧室/街道/户外），试穿结果保留原背景，不像"试衣间"，影响观感。

**方向 A（prompt 工程）：** 生成时加 `"将背景替换为干净的浅灰色摄影棚背景"`，简单但和"保留原场景"的需求互斥，需要产品层决策。

**方向 B（先抠图再生成）：** 用分割模型（如 SAM）自动抠出人物，放到干净背景上再做试穿 — 效果更可控，是商业虚拟试衣的标准路径。

**建议：** 产品默认用干净背景，高级用户可选"保留原背景"。

## 待解决架构问题

### 生成单品 vs 真实单品的入库区分

**问题：** Step 1/2 用 text prompt 生成的衣服（如米白针织衫、深咖大衣）被自动沉淀进了对话衣橱，但这些是 AI 生成物，不是用户真实拥有的衣服。

**影响：** 如果生成品和真实单品混在一个衣橱里，推荐系统会推荐用户实际没有的衣服，产品逻辑错乱。

**解决方案（代码层）：**
- 数据库 `wardrobe` 表加 `source` 字段：`real`（用户上传）/ `generated`（AI生成）
- 两条独立流程：
  - 真实入库：上传照片 → 识别 → 用户确认 → 存库（`source=real`）
  - 生成试穿：text prompt 生成 → 试穿预览 → 用户主动点"保存到衣橱"才写入（`source=generated`）
- 推荐系统默认只从 `source=real` 的单品里做搭配

## 写代码重点关注事项（Prompt 工程备忘）

> 以下是测试中发现的、写代码时必须在 prompt 层处理的问题，不处理会直接影响产品体验。

### 1. 虚拟试穿 — 基础保真约束（每次调用都要加）

```
保持人物面部特征、发型、体型、姿态完全不变。
服装单品必须严格保留原始商品特征，不允许 AI 自行美化或改色。
目标是真实试穿效果，不是时尚杂志二次设计。
不要改变鞋子。
```

**原因：** 不加会触发全局调色一致化（帽子变莫兰迪色）、鞋子被自动替换、整体"变好看"但失去保真。

---

### 2. 背景处理

```
将背景替换为干净的浅灰色摄影棚背景，光线均匀自然。
```

**原因：** 用户全身照背景杂乱，试穿结果不像试衣间。默认加，可给用户提供"保留原背景"开关。

---

### 3. 风格迁移 — 禁止 literal copy

```
参考图片仅作风格参考，禁止复用参考图中的任何具体服装设计。
所有单品必须是全新原创设计。
允许：相同风格语言 / 相同色调方向。
禁止：相同廓形、相同单品、相同图案。
保持用户原始照片的背景环境不变。
```

**原因：** 不加"禁止复用"，模型默认把参考图的衣服直接照搬到用户身上。背景也会被参考图环境 bleed。

---

### 4. 多轮变体生成 — 锚定风格边界

```
再生成一套不同的单品组合，但风格必须严格保持在 [风格标签] 范围内。
不允许改变整体色调方向。
不允许改变廓形语言。
只允许更换具体单品。
```

**原因：** 用户说"再来一个"时，模型会把"不一样"的权重排在"风格忠实"前面，导致风格漂移出边界。

---

### 5. 单品图拍摄质量提示（产品 UI 层）

用户上传单品图时必须在界面提示：
- 衣物必须完整可见，不要遮挡任何部分（影响衣长）
- 建议平铺白底拍摄（挂拍/堆放会影响品类识别准确率）
- 避免复杂背景（影响 step3 衣橱建模准确率）

**原因：** 参考图残缺 → 生成衣服偏短；非标准拍摄 → category 识别偏。这两个问题靠 prompt 解决不了，只能从源头约束输入质量。

---

### 6. 衣橱建模 — 结果置信度过滤

识别结果写入数据库前必须经过用户确认，尤其：
- 冷门品类（非标准单品如叠穿件）
- 非标准拍摄（挂拍/堆放）
- `style_tags` 建议让用户勾选确认，不直接信任模型输出

**原因：** Step 3 测试中"屁帘"被识别成"格纹吊带连衣裙"，品类完全错误但颜色/材质仍然正确。

---

### 7. 技术路径选择

| 场景 | 推荐路径 | 原因 |
|---|---|---|
| 只换上衣/下装 | Mask inpainting (D1) | 精准度最高，副作用最小 |
| 整套换装 | Image edit + 保真 prompt | 灵活，但需要完整约束 |
| 推荐新单品 | Text-only generation | edit 通道只会最小编辑，不会真正创作新单品 |
| 风格迁移 | Image edit + 禁止复用 prompt | 裸跑会 literal copy |

---

### 8. 模型选型天花板：gpt-image-2 vs 专用 VTON 模型

**核心区别：**

| 维度 | gpt-image-2（本项目） | 专用 VTON 模型（fashn.ai 等） |
|---|---|---|
| 输入形式 | 自然语言 prompt + 参考图 | 人物照片 + 服装图（两张独立输入） |
| 核心目标 | 生成视觉上合理的画面 | 只换衣服，严格保留身形/姿势/比例 |
| 底层技术 | Diffusion + 通用视觉理解 | 专用 warping + inpainting，服装贴合身体变形 |
| 服装保真（logo/图案/纹理） | ⚠️ 会被"创意改编"，含文字图案必失真 | ✅ 强，专门优化了细节保留 |
| 身形保真 | ⚠️ 可通过 prompt 约束，但仍有漂移风险 | ✅ 身形严格锁定，不会漂移 |
| 自然语言控制灵活性 | ✅ 高，直接说想要什么 | ❌ 低，只能换衣服，不能同时改场景/风格 |

**结论：**
- Step 6 的"上衣图案出错"、Step 9 的"风格 literal copy"根本原因是工具选型天花板，不是 prompt 问题
- 如果产品核心是"精准保留服装细节的虚拟试穿"（电商场景），长期应考虑接入专用 VTON 模型
- 当前用 gpt-image-2 的价值在于：灵活的风格迁移、场景生成、多轮对话——这些专用 VTON 做不到
- **两者不是替代关系，是不同场景**：精准试穿用 VTON，风格探索用 gpt-image-2
